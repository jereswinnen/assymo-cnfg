# Phase 5.8 — Poorten (gates)

**Status:** approved 2026-04-28
**Inserts between:** Phase 5.7 (Supplier catalog, shipped) and Phase 6 (Online payments, deferred)
**Parent spec:** `docs/superpowers/specs/2026-04-17-platform-architecture-design.md`

## Goal

Add **gates** as a first-class option to the configurator — buildable, configurable, priced, and shareable end-to-end — without inventing new infrastructure. Each layer slots into an existing pattern: a new building primitive (sibling to `paal`/`muur`), a new material category (sibling to `wall`/`door`), a new product kind (sibling to `overkapping`/`berging`), and a new supplier-product kind (sibling to `door`/`window`).

The result: tenants who want gates curate them per-tenant via the existing admin (materials + products + supplier products); tenants who don't want gates do nothing and the feature stays invisible. Customers either spawn a gate from a landing-page card (product) or drag one from the tray (primitive); they tweak it in the sidebar; the priced quote and order snapshot freeze it like any other line.

## Why now

Phase 5.7 finished the supplier catalog (doors + windows). The product + supplier-product machinery is in place; the configurator's draggable-primitive tray, the snap graph, the per-instance config sidebar, the per-tenant catalog gating, the order quote-snapshot freezing — all proven on doors/windows/walls. Adding gates exercises every one of these surfaces along their existing seams. Doing it now also forces the small modularity refactor (kind registry) that pays off for every future kind (poolhuis, tuinhuis, schuur-add-on, etc.) and removes hardcoded type-switches that have crept across `ObjectsTab`, `ConfigureTab`, and pricing.

## Architectural decisions

### Per-tenant feature flag = material catalog ownership

A tenant who doesn't want gates simply seeds zero materials in the `'gate'` category. With zero gate materials:

- The "Poort" entry in the configurator tray is hidden (the registry filter checks `tenantCatalog.has('gate')`).
- No `'poort'` products can be published (validator rejects products in a kind whose required categories the tenant doesn't own).
- No supplier `'gate'` products can be published either.
- The landing page shows no gate cards.

No new `tenants.flags` column, no per-feature toggle table. Material-category presence is the gate (literally), consistent with how walls/doors/windows already work. As a side benefit this gives **per-tenant primitive visibility for the existing four kinds too** — a tenant could disable `paal` by removing all wall materials, etc.

### Naming: BuildingType + ProductKind in Dutch, MaterialCategory + SupplierProductKind in English

Matches the existing convention in the codebase:

- `BuildingType` adds `'poort'` (joining `'overkapping'|'berging'|'paal'|'muur'` — all Dutch singulars).
- `ProductKind` adds `'poort'` (joining `'overkapping'|'berging'`).
- `MaterialCategory` adds `'gate'` (joining `'wall'|'roof-cover'|'roof-trim'|'floor'|'door'` — all English).
- `SupplierProductKind` adds `'gate'` (joining `'door'|'window'`).

UI display name stays "Poort" / "Poorten" (Dutch) via i18n keys, regardless of the internal identifier.

### One new SupplierProductKind: `'gate'` — covers locks, handles, colors, motors

Lock variants, handle SKUs, RAL colours, glazing, motor models all live as **option matrices on a single `'gate'` SupplierProduct**, not as separate supplier kinds. `GateMeta` carries:

```ts
type GateMeta = {
  partCount: 1 | 2 | 'configurable'
  motorized: boolean | 'optional'
  motorizedSurchargeCents?: number          // when 'optional'
  swingDirections: ('inward'|'outward'|'sliding')[]
  defaultDimensions?: { widthMm, heightMm }
  maxDimensions?:     { widthMm, heightMm }
  glazing: 'none' | 'partial' | 'full'
  availableColors?:  { sku, labelKey, ralCode?, surchargeCents? }[]
  availableLocks?:   { sku, labelKey, surchargeCents? }[]
  availableHandles?: { sku, labelKey, surchargeCents? }[]
}
```

The placed gate's `GateConfig` (on `BuildingEntity`) carries the customer's picks: `supplierProductId?`, `selectedColorSku?`, `selectedLockSku?`, `selectedHandleSku?`, `motorized`, `swingDirection`, dimensions per part. Pricing pulls base from the supplier product, sums option surcharges, adds `priceBook.poort` dials. Order submit freezes the full supplier-product row + selected options into `quoteSnapshot.items[].lineItems[i].supplierProduct` exactly like doors do today.

A "naked" gate (no `supplierProductId`, just material from the `'gate'` catalog category) is a valid configuration too — pricing falls back to per-m² material pricing + price-book surcharges only.

### Building kind registry collapses scattered type-switches into one file

Today `BuildingType` is switched on in ~6 places (`Building.tsx`, `SchematicView`, `ConfigureTab.SECTIONS`, `ObjectsTab.PRIMITIVE_ITEMS`/`STRUCTURAL_ITEMS`, `applyProductDefaults`, pricing). Adding a 5th type by adding more switches works but accelerates the rot. Introduce a single registry:

```ts
// src/domain/building/kinds.ts
export const BUILDING_KIND_META = {
  overkapping: { tray: 'structural', requiredCategories: ['wall','roof-cover','floor'], productable: true,  snapKind: 'structural' },
  berging:     { tray: 'structural', requiredCategories: ['wall','roof-cover','floor','door'], productable: true,  snapKind: 'structural' },
  paal:        { tray: 'primitive',  requiredCategories: ['wall'], productable: false, snapKind: 'pole' },
  muur:        { tray: 'primitive',  requiredCategories: ['wall'], productable: false, snapKind: 'wall' },
  poort:       { tray: 'primitive',  requiredCategories: ['gate'], productable: true,  snapKind: 'wall' },
} as const
```

Tray builders, product validators, snap rule lookups, and `applyProductDefaults` consume the registry. The R3F renderer + 2D schematic glyph + sidebar config panel stay as per-kind components — those are physical UI you cannot data-drive — but their dispatch is registry-driven. Future kinds drop in by appending a row + writing the per-kind UI components.

### Pricing split: per-m² on materials, surcharges on `priceBook.poort`

- **Per-m² material cost** lives on `MaterialPricing.gate.perSqm` (mirrors `wall.perSqm`).
- **Configuration surcharges** live on `priceBook.poort`: `motorSurchargeCents`, `slidingSurchargeCents`, `perLeafBaseCents`. Tenant-configurable in the existing admin price-book section.
- **Supplier-product option surcharges** live on `GateMeta.availableLocks[].surchargeCents` etc. — the supplier-product author owns these.
- **Quote line items** follow the existing `{ labelKey, labelParams }` shape; the UI runs `t()` at render time.

### Snap behaviour reuses the `muur` snap kernel

A 1-part gate is geometrically a `muur`; a 2-part gate is two adjacent `muur`-likes sharing a centre seam. Add `'poort'` to the snap-eligible set in `src/domain/building/snap.ts` with `snapKind: 'wall'` from the registry. A gate snaps to:

- The free edge of a `muur`
- Between two `paal`s (fills the span)
- The wall opening of a `berging` / `overkapping` (slots into a doorway-sized opening)

No new snap algorithm. The existing edge-detection handles centred 2-part gates because the gate's outer footprint is what snaps; the centre seam is a render concern, not a geometry concern.

### No new tables, no migrations

Every new piece sits in existing jsonb columns or extends an existing enum-text column whose CHECK is enforced in the application layer:

- `materials.category` → accepts `'gate'`.
- `products.kind` → accepts `'poort'`.
- `products.defaults` / `products.constraints` → extended jsonb shapes (gate-specific fields validated by domain code).
- `supplier_products.kind` → accepts `'gate'`.
- `supplier_products.meta` → accepts `GateMeta` (validated by `validateGateMeta`).
- `tenants.priceBook` → extended jsonb with optional `poort` subobject.
- `configs.data` (jsonb) → carries `BuildingEntity` rows whose `type='poort'` and `gateConfig` jsonb.

The full ledger of new code-level identifiers:

| New identifier | Type | Pattern sibling |
|---|---|---|
| `'poort'` | `BuildingType` enum value | `'muur'` |
| `'gate'` | `MaterialCategory` enum value | `'wall'` |
| `'poort'` | `ProductKind` enum value | `'overkapping'` |
| `'gate'` | `SupplierProductKind` enum value | `'door'` |
| `GateConfig` | domain struct on `BuildingEntity` | `WallConfig` |
| `GateMeta` | domain struct on `SupplierProduct` | `DoorMeta` |
| `validateGateMeta` | validator fn | `validateDoorMeta` |
| `MaterialPricing.gate` | jsonb sub-shape | `MaterialPricing.wall` |
| `priceBook.poort` | jsonb sub-shape with motor/swing dials | `priceBook.doorBase` |
| `Gate.tsx` (R3F) | per-kind 3D component | `Wall.tsx` |
| Gate schematic glyph | per-kind 2D render branch in `SchematicView` | muur glyph |
| `GateConfigPanel.tsx` | per-kind sidebar config component | wall config section |
| `BUILDING_KIND_META` | registry file | new — pays off on next kind |

Four enum extensions, two structs, one validator, one price-book subobject, three UI components, one registry. No tables, no API routes, no admin pages. Existing forms (`ProductForm.tsx`, `SupplierProductForm.tsx`, material form, price-book section) gain kind-conditional branches.

## Customer-facing flow

Two entry points, both already wired by the building blocks above:

1. **From `/`** (landing page): Click a "Poort" product card → `/configurator?product=<slug>` → `applyProductDefaults` reads the registry, finds `'poort'` is productable, hydrates one `BuildingEntity` of `type='poort'` with the product's `defaults`. Customer lands in the canvas with the gate already placed and configured.
2. **From the configurator tray**: Drag the "Poort" primitive from `ObjectsTab` → empty gate placed at cursor, customer configures from defaults. Tray entry only renders if the tenant has at least one `'gate'` material (registry filter).

In both cases, selecting a placed gate opens `GateConfigPanel` in the sidebar with these controls:

- **Parts**: radio 1 / 2 (gated by `GateMeta.partCount` if a supplier product is referenced — `'configurable'` allows both, `1` or `2` locks the choice).
- **Width / height per part**: numeric inputs, clamped by product `constraints` (if hydrated from a product) or supplier-product `maxDimensions` (if a SKU is referenced) or tenant defaults.
- **Material**: dropdown filtered to `'gate'` category via `useTenantCatalogs`.
- **Swing direction**: radio inward / outward / sliding, filtered by `GateMeta.swingDirections` if a supplier product is referenced.
- **Motorized**: toggle, gated by `GateMeta.motorized` (`true` forces on, `false` forces off, `'optional'` allows the toggle and adds `motorizedSurchargeCents`).
- **Supplier product**: optional dropdown ("Naked" / "Choose model"). When a model is selected, four extra dropdowns appear: Color / Lock / Handle / Glazing — populated from the SKU's `availableColors`/`availableLocks`/`availableHandles`.

Snap, drag, undo/redo, share-code save-and-load, order submit, quote PDF — all work unchanged because the gate is just another `BuildingEntity`.

## Admin-facing flow

- **Materials** (`/admin/catalog/materials`): existing form. The category checkboxes get one extra option ("Poort"). When checked, the pricing form reveals a `gate.perSqm` field. Texture upload unchanged.
- **Products** (`/admin/catalog/products`): existing form. The "Kind" radio gets one extra option ("Poort"). When selected, the form swaps from the building-style defaults panel (wall slots, roof slots, floor) to a gate-style defaults panel (partCount, swing, motorized, default dimensions, allowed gate materials, allowed swings, max dimensions). Implemented as a kind-conditional `<GateProductDefaults>` sub-component, mirroring how supplier-product forms branch on door vs window today.
- **Suppliers + supplier products** (`/admin/catalog/suppliers/[id]/products`): existing form. The "Kind" select gets one extra option ("Poort"). When selected, the meta form swaps to gate fields (partCount, motorized, swingDirections, glazing, default/max dimensions, availableColors/Locks/Handles repeating sub-forms). Implemented as a kind-conditional `<GateMetaForm>` sub-component, mirroring `<DoorMetaForm>` and `<WindowMetaForm>`.
- **Price book** (existing tenant price-book section): gains a "Poorten" card with three numeric inputs (`motorSurchargeCents`, `slidingSurchargeCents`, `perLeafBaseCents`).
- **Seed**: `seed.ts` extended with one example "Standaard Poort 3m enkel" product (Assymo only) so a fresh local environment shows a gate card on `/`.

No new admin pages. No new routes. Existing CRUD endpoints accept the new kinds once domain validators do.

## Sub-phases

Each independently shippable.

### 5.8.1 — Building kind registry + poort primitive + gate material category

The engine layer. After this ships, gates are a working configurator primitive — draggable from the tray, snappable, materially configurable, priced via per-m² + price-book surcharges. No products, no supplier SKUs yet.

- Add `BUILDING_KIND_META` registry in `src/domain/building/kinds.ts`. Migrate `ObjectsTab`'s `PRIMITIVE_ITEMS` / `STRUCTURAL_ITEMS` arrays to derive from the registry + tenant catalog. Migrate `applyProductDefaults` and pricing dispatch to consume the registry.
- Extend `BuildingType` with `'poort'`. Add `GateConfig` struct + defaults + canonicalisation rules. Extend `migrateConfig` (no-op for new field — defaults filled in).
- Extend `MaterialCategory` with `'gate'`. Extend `MaterialPricing` with `gate: { perSqm }`. Update material validators + admin material-form category checkbox.
- Extend `priceBook` with optional `poort: { motorSurchargeCents, slidingSurchargeCents, perLeafBaseCents }`. Update `validatePriceBookPatch` + admin price-book section.
- New `Gate.tsx` R3F component. New schematic glyph branch in `SchematicView`. New `GateConfigPanel.tsx` sidebar component (parts, dimensions, material, swing, motorized — without the supplier-product dropdown, that comes in 5.8.3).
- Extend `src/domain/building/snap.ts` to include `'poort'` with `snapKind: 'wall'`.
- Extend `calculateTotalQuote` with gate line items: per-m² from material + price-book surcharges + per-leaf base.
- i18n keys for everything (Dutch).
- Tests: domain (gate config validation, pricing, snap), smoke (R3F mount, schematic render).

### 5.8.2 — Poort ProductKind + landing card + deep-link hydration

The merchandising layer. After this ships, admin can publish gate starter kits as cards on `/`, and clicking one deep-links into a pre-configured gate.

- Extend `ProductKind` with `'poort'`. Extend `ProductDefaults` / `ProductConstraints` with gate-shaped fields (partCount, swingDirection, motorized, default dimensions, allowed swings, motorizedAllowed, allowed gate-material slugs, dimension min/max).
- Extend `validateProductCreate/Patch` with the poort branch (validates against registry's `requiredCategories`, rejects if tenant has no gate materials).
- Extend `applyProductDefaults` to hydrate a `'poort'` building from the product's defaults — uses registry to dispatch.
- Extend `ProductForm.tsx` with `<GateProductDefaults>` sub-component, kind-conditional render.
- `/` landing card already iterates products of any kind — hero image, base price, slug, dimensions all flow through unchanged.
- Seed one example "Standaard Poort 3m enkel" product for the Assymo tenant.
- i18n keys for the gate-defaults form fields.
- Tests: validators, hydration, form smoke.

### 5.8.3 — Gate SupplierProductKind + per-instance SKU/options picker

The branded-catalog layer. After this ships, admins can publish branded gate models with option matrices (lock/handle/color/glazing/motor variants), customers can pick one in the sidebar, and orders snapshot the SKU + selected options for historical stability.

- Extend `SupplierProductKind` with `'gate'`. Add `GateMeta` struct + `validateGateMeta` validator (mirrors door/window validators).
- Extend `SupplierProductForm.tsx` with `<GateMetaForm>` sub-component (partCount radio, motorized tri-state, swingDirections multi-check, default/max dimension inputs, glazing radio, repeating sub-forms for availableColors/Locks/Handles).
- Extend `GateConfigPanel.tsx` (from 5.8.1) with the supplier-product dropdown + the four conditional option dropdowns (Color/Lock/Handle/Glazing-override).
- Extend the gate placement validator with supplier-ref + dimension-fit checks (parallel to door/window): non-existent ref → `supplier_product_not_found`, archived → `supplier_product_archived`, oversized → `gate_placement_too_tall` / `_too_wide`.
- Extend pricing: when `supplierProductId` is set, base = supplier product's base price, plus selected option surcharges, plus motor surcharge if `motorized && GateMeta.motorized==='optional'`.
- Extend `quoteSnapshot.items[].lineItems[i].supplierProduct` to freeze the gate SKU + selected option SKUs at submit time.
- Extend `GET /api/shop/supplier-products?kind=gate` (the existing endpoint already takes `kind` as a query param — just accepts the new value).
- i18n keys for all option labels.
- Tests: validator, placement validator, pricing with options, snapshot.

## Out of scope

- Per-tenant feature-flag table (the material-category mechanism is the flag).
- New admin pages (existing pages absorb the new kinds via conditional sections).
- New API endpoints (existing CRUD covers the new kinds).
- Multi-leaf gates beyond 2 parts (extensible later — `partCount: 1 | 2` widens to `1 | 2 | 3 | 4` if a real customer asks).
- Gate-as-opening-on-a-wall (treating a gate as a doorway-style insertion on a `muur`/`berging` wall). Not modelled — gates are standalone primitives. If a future tenant needs "gate carved into a wall", that's a separate Phase.
- Animated gate open/close in the 3D view. The 3D component renders the gate in its closed state; swing direction is shown as an arc indicator in the 2D schematic only.
- Sliding-gate rail rendering in 3D. Same — schematic-only indicator.

## Naming check before kickoff

The values `'poort'` (BuildingType + ProductKind) and `'gate'` (MaterialCategory + SupplierProductKind) match the codebase's existing convention (Dutch for kinds that are in the Dutch domain vocabulary, English for the generic category enums). UI strings remain Dutch via i18n. Confirm before 5.8.1 kicks off; renaming any of these later is a small but cross-cutting change.
