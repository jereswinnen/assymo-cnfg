# Phase 5.5 — Catalog (Products + Materials + Pricing)

**Status:** approved 2026-04-21
**Inserts between:** Phase 5 (Invoices, shipped) and Phase 6 (Online payments, deferred)
**Parent spec:** `docs/superpowers/specs/2026-04-17-platform-architecture-design.md`

## Goal

Replace the hardcoded material registry and per-object catalog arrays with a **tenant-owned, DB-backed catalog** of Materials and Products (starter kits). Every tenant — Assymo first — authors its own catalog through the admin. The configurator reads from the catalog instead of from TypeScript constants. This is the base that makes the existing tenant feature usable for real commerce.

"Product" = a named starter kit (e.g. "Standaard Berging 3×3"), built on an engine primitive (`overkapping` or `berging`), with defaults, per-slot material constraints, and a price model. Poles (`paal`) and wall segments (`muur`) stay as engine primitives — not catalog rows — because "a pole" isn't a product you author, it's a building block.

"Material" = a tenant-authored finish/covering, categorised (wall / roof-cover / roof-trim / floor / door), with uploadable PBR textures and per-category pricing.

## Why now

Today's material set ships in code (`src/domain/materials/atoms.ts`, `src/domain/materials/catalogs/*.ts`). Changing a price, adding a finish, or onboarding a partner requires a code deploy. Phase 4.5 added a per-tenant allow-list on top of the hardcoded registry — useful but incomplete: tenants still can't add their own materials, override pricing, or upload textures. Assymo itself can't go live with the configurator for real commerce until the catalog is data-driven. This phase is that foundation.

## Architectural decisions

### One user-facing entity: Product

`overkapping | berging | paal | muur` are engine primitives — they select geometry/snap-rule kernels. Of those, only `overkapping` and `berging` are "authorable" as starter kits. `paal` and `muur` stay as tray primitives the customer can always add loose. The DB therefore has one `products` table with a `kind` column; `paal` and `muur` never appear there.

### Materials = single table discriminated by `category`

One `materials` table with a `category` column, not one table per category. Shared shape (slug, name, color, textures, tileSize) fits every category; the small category-specific differences (pricing.perSqm vs pricing.surcharge; clearsOpenings flag on walls; isVoid on floors) live in small jsonb fields validated in pure domain code. One migration, one admin UI, one API surface.

### Each tenant owns its full catalog — no inheritance

No "base catalog + overrides" model (which is where PIM systems get complicated). Every tenant's `materials` and `products` rows are theirs. Assymo is tenant #1, seeded from today's hardcoded data at migration time. Partners start from an empty catalog and author what they need. If cross-tenant sharing ever becomes a requirement, it's a later additive feature.

### Texture + hero-image storage: Vercel Blob (including Assymo)

All texture and hero-image uploads go to Vercel Blob. Assymo's seed uploads today's `/public/textures/*.jpg` files to Blob the first time `pnpm db:seed` runs, caches the returned URLs on the `materials` rows, and becomes a no-op on subsequent runs. `/public/textures/` is retired as part of 5.5.3 cleanup. One code path for texture loading (HTTPS from Blob) across every tenant.

Operational prereq: Vercel Blob store provisioned on the project; `BLOB_READ_WRITE_TOKEN` in env.

### Scalar price dials stay on `tenants.priceBook`

`insulationPerSqmPerMm`, `doorBase.{enkel,dubbel}`, `doorWindowSurcharge`, `windowFee`, `skylightFee`, `postPrice`, `bracePrice` stay where they are. Per-material €/m² and door surcharges — which currently also live on the hardcoded catalog arrays — move to `materials` rows. A post is a post regardless of product; making each Product re-specify scalar fees is authoring overhead we don't need. Hybrid (per-product override) is a future additive change if demand emerges.

### DB is effectively empty outside auth + tenants

Migrations are unconstrained — drop/recreate and destructive column changes are safe. No backfill story needed for historical data. The one exception is `tenants` and `tenant_hosts` rows, which are seeded.

### Configurator moves from `/` to `/configurator`

`/` becomes the tenant-branded landing page — a product grid + "Bouw van nul" CTA — rendered inside `<BrandedShell variant="shop">`. `/configurator` is the canvas: blank by default, pre-filled when the URL includes `?product=<slug>`. Tenants with zero products fall through from `/` directly to `/configurator` (no cold-start screen).

### Share codes move from inline base58 to server-minted short IDs (5.5.3)

Today's v6 share code encodes the full `ConfigData` as a long base58 string (slug-heavy → 100+ chars common). Replace with a server-minted `nanoid(10)` per tenant. `configs.data` (already jsonb) stays the source of truth; the `code` column holds the short ID. `POST /api/configs` dedupes by canonical content hash so the same scene always maps to the same short ID for that tenant. The pure bit-packed encoder (`encodeState` / `decodeState`) is deleted — state shape changes go through the existing `migrateConfig` path on jsonb instead of share-code versioning. No rollback concern: no production share codes exist yet.

### Shadcn is consumed in stock form

Admin UI uses shadcn primitives exactly as shipped. New primitives added via `pnpm dlx shadcn@latest add [name]`. No custom restyling of button/dialog/sidebar/table/etc. Shadcn doesn't ship a drag-and-drop primitive, so product row reordering uses `@dnd-kit/core` + `@dnd-kit/sortable` (community-standard pairing, well-maintained).

## Data model

```
materials                                -- new (5.5.1)
  id uuid PK
  tenant_id uuid FK tenants ON DELETE CASCADE
  category text                          -- 'wall'|'roof-cover'|'roof-trim'|'floor'|'door'
  slug text                              -- unique per (tenant_id, category)
  name text
  color text                             -- '#RRGGBB'
  textures jsonb                         -- { color, normal, roughness } Blob URLs, nullable
  tile_size jsonb                        -- [number, number] meters, nullable
  pricing jsonb                          -- { perSqm } | { surcharge } | {}
  flags jsonb                            -- { clearsOpenings?, isVoid? }
  archived_at timestamptz
  created_at, updated_at
  UNIQUE (tenant_id, category, slug)

products                                 -- new (5.5.2)
  id uuid PK
  tenant_id uuid FK tenants ON DELETE CASCADE
  kind text                              -- 'overkapping'|'berging' only
  slug text                              -- unique per tenant_id
  name text
  description text
  hero_image text                        -- Blob URL, nullable
  defaults jsonb                         -- { width, depth, height, walls:{…}, roof:{…}, floor:{…}, door:{…} }
  constraints jsonb                      -- { minWidth, maxWidth, …, allowedMaterialsBySlot:{…} }
  base_price_cents integer NOT NULL DEFAULT 0
  sort_order integer NOT NULL DEFAULT 0
  archived_at timestamptz
  created_at, updated_at
  UNIQUE (tenant_id, slug)

tenants.enabled_materials                -- dropped (5.5.1) — Phase 4.5 absorbed
configs.code                             -- content shifts to nanoid(10) (5.5.3)
```

`orders`, `invoices`, `payments`, `invoice_numbers`, `tenant_hosts`, `users`, `sessions`, `accounts`, `verifications` — untouched.

Buildings gain an optional `sourceProductId: string` on `BuildingEntity` so the configurator can enforce per-product constraints and expose a "Herstel standaarden" action. Null for primitives and free-build buildings. Because `sourceProductId` sits inside `BuildingEntity`, it rides along into `configSnapshot` at order submit — historical orders automatically record which Product (if any) spawned each building, no schema change on `orders`.

Framework-free (no React / Next / DB client imports) so API routes, admin, and configurator share one contract.

  material.ts                           -- validateMaterialCreate, validateMaterialPatch
  product.ts                            -- validateProductCreate, validateProductPatch,
                                        --   applyProductDefaults(product) → BuildingEntity
  constraints.ts                        -- filterMaterialsForProduct, clampDimensions

src/domain/materials/                   -- repurposed
  atoms.ts                              -- DELETED (5.5.1)
  catalogs/*.ts                         -- DELETED (5.5.1)
  resolve.ts                            -- kept; signature takes MaterialRow[] instead of closing over the registry

Validators return `{ ok: true, value } | { ok: false, errors: { field, code }[] }` — no thrown errors for user input. Error codes are stable strings (see "Error codes" below); admin UI maps them to i18n keys.

A customer browsing a partner's site must see the product grid before signing in.

## TenantContext gains `catalog`

```ts
TenantContext {
  …existing (tenantId, branding, priceBook, invoicing),
  catalog: {
    materials: MaterialRow[],
    products:  ProductRow[],    // 5.5.2+
  }
}
```

The root layout's tenant resolver fetches both in one `Promise.all` against Drizzle. Both are tens-of-rows; pagination is not a concern at this size. The catalog fetch is wrapped in React's `cache()` per-request alongside the existing tenant resolver, so layout + API routes in the same request share one query.

`useTenantCatalogs()` keeps its name and its return shape (the same five per-category buckets — wall / roofTrim / roofCover / floor / door). It gains an optional `sourceProductId` argument; when supplied, each returned bucket is narrowed to that product's `allowedMaterialsBySlot` list (empty allow-list = no filter, "all of this category is allowed"). Consumers (`BuildingMaterialSection`, `RoofConfigSection`, `FloorConfigSection`, `DoorConfig`, `SurfaceProperties`) keep their existing hook call; product-aware buildings pass the new arg through. Existing `filterCatalogAllowing(catalog, currentValue)` logic — keep the current selection visible even if it's now disallowed/archived — is retained and reused.

## Migration / seed

### Seed flow (`src/db/seed.ts`)

Idempotent, re-runnable:

The hardcoded atoms + per-object catalogs are removed from `src/domain/materials/` during 5.5.1 (framework-free domain code has no business carrying Assymo-specific content). Their content is moved verbatim to `src/db/seedData.ts` — a one-time migration payload that only the seed imports. Once every environment has seeded Assymo, `seedData.ts` can be deleted (5.5.3 cleanup); until then it's the bridge between "hardcoded TS" and "DB rows".

Seed is idempotent and re-runnable:

1. Upsert `assymo` tenant + hosts (unchanged from today).
2. For each entry in `seedData.ts` (atoms joined with their per-object catalog entries):
   - If a matching `materials` row already exists for the Assymo tenant, skip.
   - Otherwise, for each texture path, upload the `/public/textures/*.jpg` file via `@vercel/blob`'s server SDK, then insert the row with the returned Blob URLs. If `BLOB_READ_WRITE_TOKEN` is missing (local dev without Blob), skip upload and insert textures as null, logging a warning.
3. Insert two example `products` rows for Assymo — "Standaard Overkapping 4×3" and "Standaard Berging 3×3" — with permissive constraints (empty `allowedMaterialsBySlot` = all materials allowed). Illustrative; Assymo swaps them via the admin.

## Failure-mode rules

- **Delete a material referenced by a product** → 409 `catalog.material.in_use`, payload lists referencing product IDs. Admin archives instead.
- **Delete a material referenced by an existing order's `configSnapshot`** → allowed. Historical orders render from their frozen snapshot, not from foreign keys. That's why we snapshot.
- **Archived material** → hidden from pickers, remains visible as "current selection" in the configurator via `filterCatalogAllowing`.
- **Upload fails** (Blob 5xx, size/type rejected) → `catalog.upload.*` surfaced inline in the form without losing other form state.
- **Seed idempotency** → re-runs never duplicate rows; Blob upload skipped when URLs already populated.
- **Tenant with zero products** → `/` falls through to `/configurator` (blank canvas).
- **Tenant with zero materials** → configurator shows a banner pointing to `/admin/catalog/materials`; no crash.

## Out of scope

- **Cross-tenant material/product sharing or inheritance.** Additive later if needed.
- **Per-product override of scalar fees** (doorBase, postPrice, …). Additive later.
- **Tag-based material compatibility** (materials carry tags, products accept tags). Current model is explicit allow-lists; tags are a later refinement.
- **Auto-generated PBR maps from a single color texture.** Admins upload all three.
- **Admin catalog-content tooling** (CSV import/export, bulk upload). Manual authoring via forms for v1.
- **Live catalog-edit preview in the configurator** (open a material in admin → see it live in the canvas on another tab). Nice future feature.
- **Variants within a material** (e.g., "Eikenhout — licht / donker" as linked rows). Author them as distinct materials for v1.
- **Product-level add-on toggles** beyond what the engine already supports (insulation, skylight, corner braces).
