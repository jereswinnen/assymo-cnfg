# Phase 5.8.1 — Building kind registry + poort primitive + gate material category

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship `'poort'` as a working configurator primitive — draggable from the tray, snappable, materially configurable, priced via per-m² + price-book surcharges. After this phase, customers can build and price gates end-to-end against tenant-curated `'gate'` materials. No products, no supplier SKUs yet (those land in 5.8.2 and 5.8.3).

**Architecture:** A new `BuildingType: 'poort'` joins the existing four primitives. Its dispatch into 3D / 2D / sidebar / pricing / snap is mediated by a new `BUILDING_KIND_META` registry that collapses the ~6 existing hardcoded type-switches into a single source of truth. A new `MaterialCategory: 'gate'` carries gate finishes with `perSqm` pricing; the category's *presence* in a tenant's catalog is the per-tenant feature flag (zero gate materials → primitive hidden everywhere). A new `priceBook.poort` jsonb subobject holds motor / sliding / per-leaf surcharges. Snap reuses the existing `muur`-edge kernel; a 1-part gate is geometrically a wall, a 2-part gate is two adjacent walls sharing a centre seam.

**Tech Stack:** Next.js 16 (App Router) + React 19 + R3F + Drizzle ORM (Neon HTTP) + shadcn (native) + Vitest via `vite-plus/test`. Package manager: **pnpm**.

**Spec:** Parent design doc `docs/superpowers/plans/2026-04-28-phase-5.8-poorten-design.md` covers Phase 5.8 in full; this plan covers 5.8.1 only.

**Prereqs (operational, not code):**

- Worktree `.worktrees/phase-5.8.1-poort-primitive` on branch `feat/phase-5.8.1-poort-primitive` via `superpowers:using-git-worktrees` before starting.

## Architectural decisions

### The registry is the meaningful refactor

`BuildingType` is currently switched on in:
- `Building.tsx` (R3F renderer dispatch)
- `SchematicView` (2D schematic renderer dispatch)
- `ConfigureTab.tsx`'s `SECTIONS` array (`showFor` per type)
- `ObjectsTab.tsx`'s `PRIMITIVE_ITEMS` / `STRUCTURAL_ITEMS` arrays
- `applyProductDefaults` (catalog hydration)
- pricing dispatch in `calculateTotalQuote`
- snap detection in `src/domain/building/`

Adding `'poort'` via more switches works once. After that, every new kind compounds the rot. The registry replaces metadata-style switches (tray group, required material categories, productable flag, snap kind) with one data structure and leaves only the truly per-kind components (R3F renderer, schematic glyph, sidebar config panel) as switches — those have to be per-kind because they're physical UI you cannot data-drive.

```ts
// src/domain/building/kinds.ts
export const BUILDING_KIND_META = {
  overkapping: { tray: 'structural', requiredCategories: ['wall','roof-cover','floor'], productable: true,  snapKind: 'structural' },
  berging:     { tray: 'structural', requiredCategories: ['wall','roof-cover','floor','door'], productable: true,  snapKind: 'structural' },
  paal:        { tray: 'primitive',  requiredCategories: ['wall'], productable: false, snapKind: 'pole' },
  muur:        { tray: 'primitive',  requiredCategories: ['wall'], productable: false, snapKind: 'wall' },
  poort:       { tray: 'primitive',  requiredCategories: ['gate'], productable: true,  snapKind: 'wall' },
} as const

export type BuildingKindMeta = typeof BUILDING_KIND_META[BuildingType]
export const buildingKindsBy = (predicate) => …  // small helpers used by callers
```

`ObjectsTab` iterates the registry and filters tray entries by `requiredCategories.every(cat => tenantCatalog.has(cat))`. This gives **per-tenant primitive visibility for free** for the existing four kinds too — a side benefit, not the main goal, but makes the existing types properly tenant-scoped.

### Snap reuses the muur kernel — no new geometry

A 1-part gate occupies the same footprint as a `muur`. A 2-part gate is two `muur`-likes sharing a centre seam — the *outer* footprint is what snaps; the centre seam is a render concern, not a geometry concern. Add `'poort'` to the snap-eligible set in `src/domain/building/snap.ts` with `snapKind: 'wall'` from the registry. The existing edge-detection handles centred 2-part gates without modification.

### `GateConfig` lives on `BuildingEntity`, mirrors `WallConfig`

```ts
export interface GateConfig {
  partCount: 1 | 2
  partWidthMm: number          // per-part width; total = partCount × partWidthMm
  heightMm: number             // shared across parts
  materialId: string           // FK into tenant's 'gate' MaterialCategory
  swingDirection: 'inward' | 'outward' | 'sliding'
  motorized: boolean
  // 5.8.3 will add: supplierProductId?, selectedColorSku?, selectedLockSku?, selectedHandleSku?
}
```

`BuildingEntity` for `type='poort'` carries `gateConfig: GateConfig`. Position + rotation reuse the existing `BuildingEntity` shape (`position: [x, z]`, `rotation`). Total footprint is `partCount × partWidthMm` wide × ~`gateThickness` deep (small constant; gates aren't thick).

### Per-tenant flag: gate-material-category presence

A tenant who doesn't want gates seeds zero `'gate'` materials. With zero:

- Tray entry hidden (registry filter).
- Validation rejects any future `'poort'` product (5.8.2) or `'gate'` supplier product (5.8.3) at create time — the missing required-category cascades.
- No code branch for "feature disabled" — the absence of seed data is the feature flag.

### Pricing: per-m² from material + surcharges from `priceBook.poort`

```ts
priceBook.poort: {
  motorSurchargeCents: number      // applied when motorized=true
  slidingSurchargeCents: number    // applied when swingDirection='sliding'
  perLeafBaseCents: number         // applied per part (×1 for 1-part, ×2 for 2-part)
}
```

Per-gate quote line items:

```
{ labelKey: 'quote.line.gateMaterial', labelParams: { materialName, sqm }, total: areaSqm × material.pricing.gate.perSqm × parts }
{ labelKey: 'quote.line.gatePerLeaf', labelParams: { count }, total: partCount × perLeafBaseCents }   // when > 0
{ labelKey: 'quote.line.gateMotor', total: motorSurchargeCents }                                     // when motorized
{ labelKey: 'quote.line.gateSliding', total: slidingSurchargeCents }                                 // when sliding
```

Stable, deterministic, and the `{ labelKey, labelParams }` shape lets the UI run `t()` at render time per project convention.

### `MaterialCategory: 'gate'`, `MaterialPricing.gate.perSqm`

`MaterialCategory` enum gains `'gate'`. `MaterialPricing` gains `gate?: { perSqm: number }`. The shared materials table needs no schema change — `category` is text, `pricing` is jsonb, `flags` is jsonb (no new flags for gates in 5.8.1; `clearsOpenings` is meaningless for gates because gates *are* the opening).

### No DB migrations

Every new piece sits in existing jsonb columns or extends existing text-typed enums whose CHECK is application-layer:

- `materials.category` accepts `'gate'` (validator extension).
- `materials.pricing` accepts `{ gate: { perSqm } }` (validator extension).
- `tenants.priceBook` accepts optional `poort: {...}` subobject (validator extension).
- `configs.data` (jsonb) carries `BuildingEntity` rows with `type='poort'` and `gateConfig` jsonb. `migrateConfig` is a no-op for new fields (defaults filled in for legacy rows that won't have them anyway — no production poort rows exist yet).

### Out of scope (deferred to 5.8.2 / 5.8.3 / later)

- `'poort'` as `ProductKind` + landing card + deep-link hydration → 5.8.2.
- `'gate'` as `SupplierProductKind` + GateMeta + per-instance SKU/color/lock/handle picker → 5.8.3.
- Animated gate open/close in 3D.
- Gate-as-opening-on-a-wall (treating a gate as a doorway-style insertion on a `muur`/`berging`).
- `partCount > 2` (extensible later if a real customer needs 3+ leaves).
- `availableMotors` matrix (different motor brands/models). Scoped at supplier-product level in 5.8.3.

## Pricing changes

`calculateTotalQuote` gains a new dispatch via the registry:

```ts
for (const building of buildings) {
  switch (building.type) {
    case 'overkapping': … existing
    case 'berging':     … existing
    case 'paal':        … existing
    case 'muur':        … existing
    case 'poort':       items.push(...gateLineItems(building, materials, priceBook))
  }
}
```

`gateLineItems(building, materials, priceBook)` (pure, in `domain/pricing/calculate.ts`):

1. Look up `gateConfig.materialId` in `materials`. Return `material_missing` stub if not found.
2. Compute `areaSqm = (partCount × partWidthMm × heightMm) / 1_000_000`.
3. Push `{ labelKey: 'quote.line.gateMaterial', labelParams: { materialName, sqm }, total: round(areaSqm × material.pricing.gate.perSqm) }`.
4. If `priceBook.poort?.perLeafBaseCents > 0`: push `{ labelKey: 'quote.line.gatePerLeaf', labelParams: { count: partCount }, total: partCount × perLeafBaseCents }`.
5. If `motorized`: push `{ labelKey: 'quote.line.gateMotor', total: priceBook.poort?.motorSurchargeCents ?? 0 }`.
6. If `swingDirection === 'sliding'`: push `{ labelKey: 'quote.line.gateSliding', total: priceBook.poort?.slidingSurchargeCents ?? 0 }`.

Tests in `tests/pricing-gate.test.ts` cover: per-m² calculation, 1-part vs 2-part scaling, motor surcharge applied iff motorized, sliding surcharge applied iff sliding, missing material falls back gracefully, missing priceBook.poort defaults to zero surcharges.

## Notes on modularity (why this design holds up)

**Adding a 6th building type later** (e.g. `'poolhuis'`):

1. Append a row to `BUILDING_KIND_META`.
2. Write the per-kind R3F component, schematic glyph, and sidebar config panel.
3. Add the kind's required materials category (or reuse an existing one).
4. Done — no further switches to update.

**Tenant-level feature gating without a flag table:**

- The registry's `requiredCategories` declaration is the contract. Tenant catalog presence is the implementation. Adding a new "feature flag" in the future means: declare which material categories the feature requires, and the gating happens automatically.

**Splitting gate parts into separately-priced surfaces (e.g. fence panel + frame):**

- Future additive: extend `GateConfig` with a `frameMaterialId` (separate FK into a different category). `gateLineItems` adds a frame line item. No registry or schema change; pure additive in domain code.

**Gate-as-opening-on-a-wall (a hypothetical future "gate carved into a `muur`"):**

- Out of scope for 5.8.x. Would be a new mode (gate-as-opening) parallel to today's gate-as-primitive, modelled the way doors are today (an opening on `WallConfig` with `gateSupplierProductId`). The supplier-product groundwork from 5.8.3 makes this a small additive lift if it's ever needed.
