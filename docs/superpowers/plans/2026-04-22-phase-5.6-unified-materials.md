# Phase 5.6 — Unified materials (retroactive plan)

**Status:** shipped 2026-04-22 in commit `aa4bf0b`. This document was written after-the-fact to close the gap in the Progress section; every other sub-phase has a linked plan. Content reconstructed from the commit + migration + diff; no new design work.

**Goal:** collapse the per-`(tenant, category, slug)` material model into a per-`(tenant, slug)` model where each row carries a `categories` array and a per-category `pricing` map. A single "Hout" row now serves wall + door + roof-trim with one texture upload and per-slot pricing, instead of three separate rows with duplicated textures and slug collisions.

**Architecture:** `materials.category` (scalar) becomes `materials.categories text[]`. `materials.pricing` shape shifts from a flat `{ perSqm }` / `{ surcharge }` to a per-category map: `{ wall?: { perSqm }, door?: { surcharge }, … }`. Uniqueness moves from `(tenant_id, category, slug)` → `(tenant_id, slug)`. Validators + admin UI + consumer call-sites flow through.

---

## Motivation

Pre-5.6: authoring "Hout" for wall + door + roof-trim required 3 rows, 3 texture uploads, 3 slugs fighting for namespace within each category. Slug collisions across categories were tolerated but confusing (`wood` for wall material ≠ `wood` for door material, yet both shared the admin list view). Products referencing materials had to track `(slug, category)` tuples.

Post-5.6: one "Hout" row with `categories: ['wall','door','roof-trim']` and pricing `{ wall: { perSqm: 45 }, door: { surcharge: 20 } }`. One texture upload. One slug per tenant. Products still reference by slug; category is resolved at render time.

---

## Data model

**`materials` table — before:**

```
tenant_id, category text, slug, name, color, textures jsonb, tile_size jsonb,
pricing jsonb,   -- flat: { perSqm } | { surcharge } | {}
flags jsonb, archived_at
UNIQUE (tenant_id, category, slug)
```

**`materials` table — after (migration 0013):**

```
tenant_id, categories text[] NOT NULL DEFAULT '{}',
slug, name, color, textures jsonb, tile_size jsonb,
pricing jsonb,   -- per-category map, e.g.
                 --   { wall: { perSqm }, door: { surcharge }, floor: { perSqm } }
flags jsonb, archived_at
UNIQUE (tenant_id, slug)
```

The `category` scalar column is dropped. `categories` is backfilled from the scalar, then duplicate `(tenant_id, slug)` rows are merged (union categories, merge pricing, prefer non-null textures + tileSize, keep the oldest `created_at` as the representative).

---

## Migration 0013 (hand-audited)

Seven numbered steps, each `--> statement-breakpoint`-delimited:

1. Add `categories text[]` nullable.
2. Backfill `categories = ARRAY[category]`.
3. Rewrite `pricing` from flat to per-category map (scoped by the existing `category` scalar).
4. Collapse duplicate `(tenant_id, slug)` rows via a PL/pgSQL `DO $$ … $$` block that picks a representative per group, unions categories, merges pricing, and deletes siblings.
5. Drop `materials_tenant_category_slug_idx`.
6. Create `materials_tenant_slug_idx` (UNIQUE on the new pair).
7. Drop the `category` column.

Product references by `(slug, category)` remain valid: slugs stay stable across the collapse; categories union rather than narrow.

## Domain changes (`src/domain/catalog/`)

- `types.ts` — `MaterialRow` gains `categories: MaterialCategory[]` (was `category: MaterialCategory`); `MaterialPricing` becomes `{ [K in MaterialCategory]?: PricingEntry }` with kind-specific entries.
- `material.ts` — `validateMaterialCreate` + `validateMaterialPatch` rewritten:
  - `categories` must be a non-empty array of valid `MaterialCategory` values.
  - Every pricing-bearing category present in `categories` must have a matching pricing entry.
  - `flags.clearsOpenings` requires `'wall'` to be in `categories`; `flags.isVoid` requires `'floor'`.
  - New stable error codes: `categories_invalid`, `categories_empty`, `pricing_category_mismatch`.
- `constraints.ts` — product-material coherence checks operate on `categories.includes(cat)` instead of `category === cat`.

## Consumers

All call-sites that previously filtered by `material.category === cat` now use `material.categories.includes(cat)`:

- `src/domain/materials/filter.ts` — category-filtered views.
- `src/lib/useTenantCatalogs.ts` — per-category memoised catalogs.
- `src/lib/productRefCheck.ts` — validates a product's `allowedMaterialsBySlot` slugs actually exist in the matching category.
- `src/db/resolveTenant.ts`'s `materialDbRowToDomain` — direct field passthrough (categories stays an array).

Canvas + schematic rendering hooks (`useWallTexture`, `useDoorTexture`, `useRoofTexture`) already took `materialId` + category as inputs; no signature change needed.

---

## Admin UI (`src/components/admin/catalog/`)

- `MaterialForm.tsx` — rebuilt:
  - Category selection becomes a checkbox group (was a single Select).
  - Pricing inputs render conditionally: per selected category, show the right field (`€/m²` for wall/roof-cover/floor, `surcharge €` for door).
  - Flag toggles (`clearsOpenings`, `isVoid`) render only when their requisite category is selected.
- `MaterialsTable.tsx` — the "category" column becomes a comma-separated list of categories; `formatPrice` summarises the per-category map.
- `ProductForm.tsx` — tiny change: allow-list lookups walk through `material.categories.includes(slot)` instead of equality.

## Notes for future work

- Phase 5.7 (supplier catalog) was the immediate follow-up; it builds **alongside** `materials` — suppliers + supplier_products are separate tables, not a re-use of the unified material shape.
- If a third "pricing shape" appears (e.g. `{ perLinearMeter }` for trims), extend `MaterialPricing` as an additional optional key per category. The schema's `pricing jsonb` already accommodates arbitrary shapes; only the validators + form + renderer need touches.
- Row-ownership-as-allow-list (existence of a row implies "available") replaced Phase 4.5's explicit `tenants.enabled_materials` allow-list — that column was dropped back in Phase 5.5.1. No re-introduction needed now.
