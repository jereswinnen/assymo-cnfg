# Phase 5.5.3 — Hardening + Share-Code Refactor + Cleanup Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close out the Phase 5.5 umbrella by (1) retiring the legacy building defaults (`DEFAULT_BUILDING`, `getDefaultWalls`, `DEFAULT_WALL`, `DEFAULT_ROOF` — keep only what product-defaults or explicit engine constants still need), (2) replacing the long bit-packed base58 share code (`encodeState` / `decodeState` / `BitWriter` / `BitReader`) with a server-minted `nanoid(10)` plus content-hash dedup on `POST /api/configs`, (3) deleting the one-time migration bridge (`src/db/seedData.ts` + `/public/textures/*`) now that the DB is the source of truth, and (4) polishing the admin catalog UX (restore buttons, duplicate action, bulk archive, empty states, i18n keys, stock shadcn `Select`, dead-code sweep).

**Architecture:** The share-code refactor is the highest-risk piece. `configs.data` (jsonb) already stores the canonical `ConfigData` — that stays the source of truth. The `configs.code` column shifts from "round-trippable bit-packed encoding" to "opaque short ID minted by the server at save time". `POST /api/configs` accepts `ConfigData` directly in the body (not a code string), canonicalises it via sorted-key JSON stringification, computes a SHA-256 content hash, and either returns the existing row for that `(tenantId, contentHash)` pair or inserts a new row with a fresh `nanoid(10)` from a base58-safe alphabet. A new `content_hash` column (unique per tenant) drives the dedup; the alphabet matches today's base58 charset so admins migrating between the old and new formats see familiar-looking codes. `migrateConfig` continues to run on `ConfigData` at read time. `useSubmitOrder` posts raw `ConfigData` (not `encodeState(…)` output), then chains `POST /api/shop/orders` with the returned short code. Legacy defaults (`DEFAULT_WALL`, `DEFAULT_ROOF`, `getDefaultWalls`) are reshaped: the single remaining fallback (walls mutation adding a wall that didn't exist) uses a blank `WallConfig` literal; `DEFAULT_ROOF` inlines into `makeInitialConfig`; `getDefaultWalls` inlines into `createBuilding`. This removes the three symbols + `src/db/seedData.ts` + `/public/textures/` + hundreds of lines of `codec.ts`. Admin polish leans on shadcn `Select` (already installed) and existing restore endpoints.

**Tech Stack:** Next.js 16 (App Router) + React 19 + Drizzle ORM (Neon HTTP) + shadcn (native) + Vitest via `vite-plus/test` + `nanoid`. Package manager: **pnpm**.

**Spec:** `docs/superpowers/specs/2026-04-21-phase-5.5-catalog-design.md` (Phase 5.5.3 section of "Sub-phase rollout order" + "Share codes move from inline base58 to server-minted short IDs (5.5.3)").

**Prereqs (operational, not code):**
- Phases 5.5.1 and 5.5.2 merged to `main` (catalog DB + products + landing live).
- Worktree + branch `feat/phase-5.5.3-cleanup` via `superpowers:using-git-worktrees` before starting. `.env.local` symlinked into the worktree for DB commands.
- No existing share-code rollback concern — `configs` rows in the DB are effectively test data. The migration drops and recreates them (see Task 14).

**Scope locks (from the umbrella spec):**
- `paal` / `muur` stay engine primitives (no changes).
- `configs.data` stays the source of truth. `configs.code` becomes an opaque short ID.
- `migrateConfig` + `validateConfig` are retained — they operate on `ConfigData`, not on the code string.
- Admin UI stays on stock shadcn primitives.
- DB is effectively empty outside auth + tenants + catalog seed — destructive migrations are pre-approved.
- Net-new lint errors only are a blocker.

```ts
import type { ConfigData } from './types';

/** Stable JSON-stringify with sorted object keys at every level, so two
 *  ConfigData values that are structurally equal always produce the
 *  same string. Used as the pre-image for `contentHash`. */
export function canonicalizeConfig(cfg: ConfigData): string {
  return stringifyStable(cfg);
}

function stringifyStable(v: unknown): string {
  if (v === null || typeof v !== 'object') return JSON.stringify(v);
  if (Array.isArray(v)) return '[' + v.map(stringifyStable).join(',') + ']';
  const keys = Object.keys(v as Record<string, unknown>).sort();
  return (
    '{' +
    keys
      .map((k) => JSON.stringify(k) + ':' + stringifyStable((v as Record<string, unknown>)[k]))
      .join(',') +
    '}'
  );
}

/** SHA-256 of the canonicalised config, hex-encoded. Used to dedupe
 *  identical scenes inside `POST /api/configs` (unique index on
 *  `(tenant_id, content_hash)`). Web Crypto is available in both Node 20+
 *  and the Edge runtime; no Node-only imports. */
export async function contentHash(cfg: ConfigData): Promise<string> {
  const bytes = new TextEncoder().encode(canonicalizeConfig(cfg));
  const digest = await crypto.subtle.digest('SHA-256', bytes);
  return Array.from(new Uint8Array(digest))
    .map((b) => b.toString(16).padStart(2, '0'))
    .join('');
}
```

Replace the whole file with:

```ts
import { NextResponse, type NextRequest } from 'next/server';
import { and, eq } from 'drizzle-orm';
import { customAlphabet } from 'nanoid';
import { db } from '@/db/client';
import { configs } from '@/db/schema';
import {
  contentHash,
  migrateConfig,
  validateConfig,
  type LegacyConfig,
} from '@/domain/config';
import { resolveApiTenant } from '@/lib/apiTenant';

/** Bitcoin-style base58 (no 0/O/I/l). 10 chars ≈ 57.5 bits of entropy.
 *  Collision probability < 1e-8 per tenant at 10k saved scenes. */
const BASE58 = '123456789ABCDEFGHJKLMNPQRSTUVWXYZabcdefghijkmnopqrstuvwxyz';
const newCode = customAlphabet(BASE58, 10);

/** Save a configurator scene. Dedupe per tenant by SHA-256 content hash
 *  over the canonicalised ConfigData — saving the same scene twice
 *  returns the existing short code. */
export async function POST(req: NextRequest) {
  const tenant = await resolveApiTenant();
  if (!tenant) {
    return NextResponse.json({ error: 'unknown_tenant' }, { status: 404 });
  }

  let body: { data?: unknown };
  try {
    body = (await req.json()) as { data?: unknown };
  } catch {
    return NextResponse.json({ error: 'invalid_json' }, { status: 400 });
  }

  if (!body.data || typeof body.data !== 'object') {
    return NextResponse.json({ error: 'data_required' }, { status: 400 });
  }

  let migrated;
  try {
    migrated = migrateConfig(body.data as LegacyConfig);
  } catch {
    return NextResponse.json({ error: 'invalid_data' }, { status: 400 });
  }

  const errors = validateConfig(migrated);
  if (errors.length > 0) {
    return NextResponse.json(
      { error: 'validation_failed', details: errors },
      { status: 422 },
    );
  }

  const hash = await contentHash(migrated);

  const [existing] = await db
    .select({ id: configs.id, code: configs.code })
    .from(configs)
    .where(and(eq(configs.tenantId, tenant.id), eq(configs.contentHash, hash)))
    .limit(1);

  if (existing) {
    return NextResponse.json({ id: existing.id, code: existing.code }, { status: 200 });
  }

  const id = crypto.randomUUID();
  const code = newCode();
  await db.insert(configs).values({
    id,
    tenantId: tenant.id,
    code,
    contentHash: hash,
    data: migrated,
    version: migrated.version,
  });

  return NextResponse.json({ id, code }, { status: 201 });
}
```

async function saveConfig(
  data: ConfigData,
  fetchFn: NonNullable<SubmitOrderInput['fetch']> = globalThis.fetch,
): Promise<{ kind: 'ok'; id: string; code: string } | { kind: 'error'; code: string; details?: string[] }> {
  try {
    const { status, data: resBody } = await postJson<{ id: string; code: string }>(
      fetchFn,
      '/api/configs',
      { data },
    );
    if (status === 200 || status === 201) {
      const ok = resBody as { id: string; code: string };
      return { kind: 'ok', id: ok.id, code: ok.code };
    }
    const err = resBody as ErrorResponseBody;
    return {
      kind: 'error',
      code: typeof err.error === 'string' ? err.error : 'unknown',
      details: Array.isArray(err.details)
        ? err.details.filter((d): d is string => typeof d === 'string')
        : undefined,
    };
  } catch (e) {
    if (e instanceof TypeError) return { kind: 'error', code: 'network' };
    return { kind: 'error', code: 'unknown' };
  }
}

async function submitOrderFromCode(
  input: Omit<SubmitOrderInput, 'code'> & { code: string },
): Promise<SubmitOrderResult> {
  const fetchFn = input.fetch ?? globalThis.fetch;
  try {
    const { status, data } = await postJson<{
      id: string;
      status: string;
      totalCents: number;
      currency: string;
      emailDispatched: boolean;
    }>(fetchFn, '/api/shop/orders', {
      code: input.code,
      contact: input.contact,
    });
    if (status === 201) {
      const ok = data as {
        id: string;
        totalCents: number;
        currency: string;
        emailDispatched: boolean;
      };

        kind: 'success',
        orderId: ok.id,
        totalCents: ok.totalCents,
        currency: ok.currency,
        emailDispatched: ok.emailDispatched,

      code: typeof err.error === 'string' ? err.error : 'unknown',
      details: Array.isArray(err.details)
        ? err.details.filter((d): d is string => typeof d === 'string')
        : undefined,
    };
  } catch (e) {

## Notes on the share-code refactor

Short rationale captured here so future-me doesn't re-derive it:

- **Why server-minted, not client-minted?** Client nanoid would skip collision detection per tenant. A server mint + unique constraint keeps the table small (≤ tens of thousands of rows per tenant) without per-client retry logic.
- **Why content hash, not a naïve "insert-and-dedupe"?** Saving the same scene twice in a row is the common case (admin shares, user copies). Content-hash dedup makes the two-POST submit flow idempotent without a second round-trip to test for duplicates. SHA-256 is overkill for collision purposes but hashing is cheap (< 1 ms) and the stable pre-image (canonicalised JSON) makes the behaviour transparent.
- **Why 10 chars of base58?** 58^10 ≈ 4.3e17 possible codes — collision probability per tenant is negligible at realistic sizes (5e-11 at 10k entries). Shorter than 10 (say 8) gives acceptable entropy but triples the lookup-duplicate probability at a million entries across all tenants; longer than 10 loses the "types on an SMS" affordance. 10 was chosen to match the legacy "group into 4-char chunks" readability.
- **Why destructive migration?** `configs` holds test scenes only — no customer orders persist directly from it (orders freeze their own snapshot). Per project convention, the DB is considered empty outside auth + tenants + catalog seed, so drop + recreate is the simplest correct migration. If a partner later needs to preserve their saved shapes, they re-save.
