# Phase 5.7 — Supplier catalog (doors + windows) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Let tenants author a supplier catalog of **specific, fixed-size door and window products** (with SKU, unit price, hero photography, and kind-specific metadata) as a parallel concept to finish materials. The configurator gains a "Eigen keuze ↔ Uit catalogus" fork per door and per window — customers can either design with the existing material-based approach OR pick a pre-packaged supplier product as-is. Orders snapshot the supplier product at submit time so pricing is stable against catalog drift.

**Architecture:** One polymorphic `supplier_products` table with a `kind` discriminator (`door | window`) + kind-specific `meta` jsonb. One `suppliers` table for the brand + contact envelope. Supplier products are **leaves**, not compositions (contrast with `products`, which are starter-kit compositions). The configurator's door/window config panels gain a mode toggle; when a supplier product is selected, dimensions + imagery + price come from the product row, and the material/size inputs lock. The `calculateTotalQuote` pricing engine gets a single new branch per surface (door or window): if that surface carries a `supplierProductId`, the line item becomes a unit-price pull from the product; else the existing material-based calculation runs. Same snapshot pattern as materials: `configSnapshot` freezes `supplierProductId` + unit price + dimensions at submit time. Canvas renders supplier products as a flat-photo material on a fixed-dimension mesh (no PBR maps — hero image only). Design is deliberately modular: adding a third product kind (shutters, ventilation, lights, anything else a supplier might sell) is schema-free (extend the kind enum + add a validator + add an admin subtype).

**Tech Stack:** Next.js 16 (App Router) + React 19 + Drizzle ORM (Neon HTTP) + shadcn (native) + Vitest via `vite-plus/test` + `@vercel/blob` (supplier hero images). Package manager: **pnpm**.

**Spec:** Self-contained (this file). Links into the umbrella `docs/superpowers/specs/2026-04-17-platform-architecture-design.md` as the Phase 5.7 entry between Phase 5.6 and Phase 6.

## Architectural decisions

### Supplier products are leaves, not compositions

`products` (starter kits in Phase 5.5.2) are compositions of primitives, with dimension defaults + material allow-lists. `supplier_products` are **single-SKU leaves** — one row = one sellable article with fixed width/height, fixed price, and one hero photograph. They never compose into anything; they're picked as-is by the customer.

Keeping them in a separate table preserves the distinction. Overloading `products` would bloat its schema (SKU, unit price, fixed dimensions, supplier_id — all null for 95% of rows) and confuse the admin mental model.

### One table per discriminator, not per kind

`supplier_products` with `kind: 'door' | 'window'` + kind-specific `meta jsonb`, rather than separate `supplier_doors` + `supplier_windows` tables. Reasons:

- **Modular.** Adding a third kind (shutter, ventilation grille, skylight window, automated door) = add an enum value + a validator + an admin subtype. No migration.
- **Shared shape is real.** Every kind needs: tenant_id, supplier_id, sku, name, hero_image, width_mm, height_mm, price_cents, sort_order, archived_at. Only ~2-3 attributes per kind are genuinely kind-specific (u_value + glazingType for windows, swingDirection + lockType for doors).
- **Simpler queries.** One list view, one admin CRUD. Kind-specific fields live inside `meta` and are validated by kind-specific validators in pure domain code.

Trade-off: the `meta jsonb` is untyped at the DB level. Validators enforce shape; the TS `SupplierProduct<Kind>` discriminated union recovers type safety at the app layer.

### Suppliers as a first-class entity

Even with one tenant signing one supplier deal initially, name the concept now. `suppliers` is a small table (name, slug, logo_url, contact jsonb, notes, archived_at). Every `supplier_products` row has `supplier_id` NOT NULL — no orphaned products. Future features like "list all products from supplier X" or "negotiate supplier-level discounts" are just queries; no refactor needed.

### Imagery: hero photograph, no PBR

Supplier photography is usually product shots (front elevation of the door/window as it ships). Those aren't seamless, tileable PBR textures — they're photographs of a specific-size object. Rendering approach:

- **3D canvas:** use the hero image as a **flat, fit-to-geometry texture** on the door/window mesh. The mesh size matches the supplier-declared width + height. `MeshStandardMaterial` with the hero image as `map`, `tileSize` = `[width_m, height_m]`, `repeat.set(1, 1)`.
- **No normal/roughness maps** unless the supplier ships them (rare). Default roughness ≈ 0.6, metalness 0.
- **Fallback:** if `hero_image` is null, show a solid-color placeholder (the supplier's brand color, or neutral gray).

This isn't photoreal. It's acceptable because it's a photograph of a real thing — the tradeoff most admins will make vs. the cost of PBR-authoring per SKU.

Future polish (out of scope): optional glTF upload per product for full 3D fidelity. Only worth adding when suppliers provide glTF.

### Pricing: unit price replaces material+surcharge when a supplier product is selected

When a wall's `doorSupplierProductId` is set, the quote's door line item becomes `{ supplierProductId, priceCents }` — pulled verbatim from the supplier product row. The existing `priceBook.doorBase` + `materialSurcharge` + `doorWindowSurcharge` chain does NOT run for that door.

Window line items: same treatment. When a window's `supplierProductId` is set, the `windowFee` from priceBook is replaced by the supplier's `priceCents`.

Quote line items gain a `source` tag so the PDF + viewer can label them distinctly:

```ts
type LineItemSource =
  | { kind: 'priceBook'; dial: string }   // e.g. postPrice, bracePrice
  | { kind: 'material'; slug: string; category: MaterialCategory }
  | { kind: 'supplierProduct'; productId: string; sku: string };
```

### Snapshotting

Orders at submit time freeze:

- `configSnapshot.buildings[].walls[side].doorSupplierProduct?: { id, sku, name, width, height, priceCents }` — denormalized copy of the product row.
- Windows: `walls[side].windows[i].supplierProduct?: { id, sku, name, width, height, priceCents }`.

Historical orders render from the snapshot. Supplier catalog drift (new prices, renamed SKUs, archived products) doesn't affect what the customer agreed to pay.

### Geometry validation

Supplier-declared dimensions are authoritative, but must still fit the surrounding building. When a supplier product is selected:

- **Door/window height vs wall height:** if `height_mm / 1000 > wall.height`, the configurator shows an inline warning on the door/window config card ("Deur is 2400 mm hoog — muur is slechts 2200 mm. Kies een andere bouwset of lagere deur."). The selection is still allowed (so the customer sees the pick + price), but submit-order is blocked until resolved. Validator lives in `src/domain/supplier/product.ts` → `validatePlacement({ wall, product })` returning `{ ok: true } | { ok: false, code: 'too_tall' | 'too_wide' }`.
- **Door width vs wall width:** `width_mm / 1000 + 2 × EDGE_CLEARANCE` must fit within `wall.length`. Same warning + submit-block pattern.
- **Window width vs wall width:** same treatment.
- **Overlapping openings:** the existing `clampOpeningPosition` helper already handles this for generic doors/windows; supplier dimensions flow through the same path so the clamp works identically.

These checks are client-side hints for UX, **and** server-side guards on `POST /api/shop/orders` so a submit can't slip through with an impossible geometry. Snapshots capture the product even when the placement is invalid (so the customer's rejected order still records what they tried to do), but the order status stays `draft` / rejected with a validation-error response.

### Out of scope (deferred)

- **Hardware bundles** (handle + lock + hinge sets sold together). Additive later; one row per physical article, composed via a future `bundles` concept.
- **Live supplier feeds** (API pulls from Weekamp/Reynaers/VELUX etc.). Admin manual entry only for v1.
- **Per-product PBR textures or glTF uploads.** Flat hero photograph only.
- **Variant matrices** (one model in 8 colours × 4 sizes = 32 rows). Admin authors each as its own row for v1; auto-matrix creation is a later polish.
- **Supplier-specific pricing rules** (volume discounts, bulk surcharges, seasonal adjustments). Fixed unit price per product.
- **Lead time / stock tracking.** No inventory layer — every supplier product is assumed available. `meta.leadTimeDays` can be informational but doesn't gate the order.

## Data model

```
suppliers                                -- new
  id uuid PK
  tenant_id text FK tenants ON DELETE CASCADE
  slug text                              -- unique per tenant, admin-visible
  name text                              -- display name
  logo_url text                          -- Blob URL, nullable
  contact jsonb                          -- { email?, phone?, website? }, validated
  notes text                             -- internal admin notes, nullable
  archived_at timestamptz
  created_at, updated_at
  UNIQUE (tenant_id, slug)

supplier_products                        -- new, polymorphic leaves
  id uuid PK
  tenant_id text FK tenants ON DELETE CASCADE
  supplier_id uuid FK suppliers ON DELETE RESTRICT
  kind text                              -- 'door' | 'window' (initial);
                                         -- extensible enum
  sku text                               -- unique per (tenant_id, kind)
  name text                              -- e.g. "Weekamp M45 Eiken"
  hero_image text                        -- Blob URL, nullable (shows placeholder)
  width_mm integer NOT NULL              -- millimeters (precision matters for
                                         -- real-world door/window sizes)
  height_mm integer NOT NULL
  price_cents integer NOT NULL DEFAULT 0
  meta jsonb                             -- kind-specific, validated by kind
  sort_order integer NOT NULL DEFAULT 0
  archived_at timestamptz
  created_at, updated_at
  UNIQUE (tenant_id, kind, sku)
  INDEX (tenant_id, supplier_id)
  INDEX (tenant_id, kind) for picker queries

BuildingEntity changes (jsonb on orders.configSnapshot):
  walls[side].doorSupplierProductId?: string | null
  walls[side].windows[i].supplierProductId?: string | null
```

`meta` shapes per kind (validated in `src/domain/supplier/product.ts`):

```ts
// door
interface DoorMeta {
  swingDirection?: 'inward' | 'outward' | 'none';
  lockType?: 'cylinder' | 'multipoint' | 'none';
  glazing?: 'solid' | 'glass-panel' | 'half-glass';
  rValue?: number;              // insulation, W/m²K
  leadTimeDays?: number;        // informational
}

// window
interface WindowMeta {
  glazingType?: 'double' | 'triple' | 'single';
  uValue?: number;              // thermal transmittance
  frameMaterial?: string;       // free text, e.g. "aluminium grijs"
  openable?: boolean;
  leadTimeDays?: number;
}
```

Both carry stable error codes (`supplier.product.meta_invalid`, `supplier.product.dimensions_invalid`, etc.).

## TenantContext gains `supplierProducts`

Same pattern as materials + products: the root layout resolves tenant → cached fetch of supplier products → passes into `TenantProvider`. Tens-of-rows scale, one request shared via React `cache()`.

Configurator consumers read from context — no client fetches. Door/window configs derive per-kind lists via `supplierCatalog.products.filter(p => p.kind === 'door')`.

## Configurator integration

### DoorConfig.tsx (existing)

Current shape: toggle `hasDoor`, pick material override, pick size preset (enkel/dubbel), position slider, swing picker.

Add at the top of the door's config card a **mode toggle** (shadcn `Tabs` or `ToggleGroup`):

```
[ Eigen keuze ]  [ Uit catalogus ]
```

- **Eigen keuze** (default): current flow. `wallCfg.doorSupplierProductId === null`.
- **Uit catalogus**: shows a supplier picker — a scrollable list of door products (`kind: 'door'`) as `Card`s with hero image, name, supplier name, dimensions, price. Click to select. Selection sets `wallCfg.doorSupplierProductId` and locks the material/size inputs (visually grayed-out with an info row: *"Afmetingen bepaald door {supplier}: 1600 × 2100 mm"*).

On switching modes, the previous mode's state is preserved (zustand's merge semantics). Deselecting a supplier product in "Uit catalogus" clears `doorSupplierProductId` but leaves the "Eigen keuze" fields as they were.

### WindowConfig.tsx

Same pattern. Add a "Uit catalogus" tab to each window's config block. Supplier window selection locks width/height/sillHeight to the supplier's values.

### Canvas rendering

`DoorMesh.tsx`: if `doorSupplierProductId` is set:
- Look up `supplierProduct` in TenantContext.
- Geometry uses `product.width_mm / 1000` × `product.height_mm / 1000` instead of the computed door-size preset.
- Material uses hero image as a flat `map` (no tiling, `repeat.set(1, 1)`), fallback to solid color + supplier logo mark when `hero_image` is null.
- No normal or roughness maps.

`WindowFrame.tsx`: equivalent for windows.

### Schematic rendering

Top-down SVG. The supplier's `width_mm / 1000` is **authoritative for the opening width** — the door slot or window opening in the wall line is drawn at the supplier's exact dimension, positioned by the usual `doorPosition` / `windowPosition` fraction. The 2D and 3D views must agree on real-world dimensions; a Reynaers 900mm door is 900mm wide in both.

A small supplier label (name or SKU) renders next to the opening for recognizability, so the admin / customer can tell which product is placed where when the plan has multiple openings.

### Quote summary

`QuoteSummary.tsx`: line items already structured. Add a small source indicator (e.g. icon or supplier-name suffix) on supplier-sourced lines so the customer sees "Deur — Weekamp M45 Eiken €650" instead of just "Deur (hout, enkel) €280".

## Pricing changes

`calculateTotalQuote` gains **one new argument**: `supplierProducts: SupplierProductRow[]`. It's passed through from the tenant resolver same as materials.

For each wall with a door:
- If `wall.doorSupplierProductId != null`:
  - Look up the supplier product.
  - Push `{ labelKey: 'quote.line.supplierDoor', labelParams: { name, sku }, total: priceCents, source: { kind: 'supplierProduct', productId, sku } }`.
  - Skip material-surcharge calculation for that door.
- Else: existing material-based calculation.

Same for windows.

Unit tests in `tests/pricing-supplier.test.ts` cover: supplier-door replaces material-surcharge, supplier-window replaces windowFee, missing product ID falls back gracefully (renders a stub line item with `labelKey: 'quote.line.supplierMissing'`), archived product still snapshots its last known price on order submit.

## Notes on modularity (why this design holds up)

**Adding a third product kind** (e.g. "shutter"):
1. Extend `SupplierProductKind` enum: `'door' | 'window' | 'shutter'`.
2. Add `ShutterMeta` interface + `validateShutterMeta` in `src/domain/supplier/product.ts`.
3. Add a new admin form subtype + configurator integration point (wherever shutters are placed — likely a new property on walls or a standalone primitive).
4. No DB migration. No schema churn. No impact on doors + windows.

**Adding a second supplier** for the same tenant:
1. Create via admin. One row in `suppliers`.
2. Author products against that supplier. The `supplier_id` column is enough — the configurator shows products grouped by supplier in the picker (or mixed, sorted by `sort_order`).
3. Customers see all active products regardless of supplier.

**Supporting supplier-level settings** (volume discounts, delivery terms, exclusive deals):
1. Add columns to `suppliers` (or a `supplier_settings` jsonb if many).
2. `calculateTotalQuote` reads them when applying supplier product pricing.
3. No impact on product structure.

**Renaming / archiving a supplier:**
- Archived supplier: flip `archived_at`; its products auto-hide from the public picker (via `archived_at IS NULL` + supplier non-archived joint filter).
- Renamed supplier: update the single row; historical orders still show the snapshot name.

**Migration to per-product glTF models (future):**
- Add `model_url text` column to `supplier_products`.
- Canvas renders glTF when present, falls back to flat hero when null.
- Additive. No customer-facing change for existing products.

## Tradeoff worth documenting upfront

The `meta jsonb` carries kind-specific fields without DB-level type safety. Admin validators enforce shape; TS discriminated unions recover it at the app layer. This is a conscious choice — the alternative (one table per kind) scales poorly past 2 kinds and blocks the modularity argument above. If a kind's `meta` ever needs to participate in queries (e.g. "filter windows by uValue < 1.2"), it'd need a generated column or a dedicated index on the jsonb path. That's a later additive change.
