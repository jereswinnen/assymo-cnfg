# Phase 6.5 — AI-assisted catalog ingest (design addendum)

**Status:** design, not yet scheduled. Addendum to the Phase 6 (Online payments) track — AI features should land after Phase 6 is stable since they're additive, not on the critical commerce path.

**Goal:** let tenants upload a supplier's price list (PDF / XLSX / CSV) and get a reviewed, editable import queue of supplier products — collapsing a manual data-entry job of hours into a review cycle of minutes. First feature in the broader AI track (see "Forward-look" at the bottom).

**Non-goals (v1):**
- No auto-import. Every extracted row passes through human review.
- No product (starter-kit) or material ingest — only supplier products. Different shape, different stakes, different tuning.
- No multi-supplier detection within a single upload. "One upload = one supplier".
- No real-time streaming of extracted rows in the UI. Async job + poll or SSE.

---

## Motivation

Supplier catalogs arrive as PDFs. Today a tenant authoring 200 doors spends a day on data entry: SKU, dimensions, price, kind, meta — all manually typed into `SupplierProductForm`. The shape of the data is deterministic; the friction is pure transcription. LLMs handle this class of task well *if* the app keeps them on a short leash.

**Claim:** for supplier products specifically, an LLM can draft the rows, a human can review them in ~10% of the original time, and the validator layer catches the worst mistakes.

**Why supplier products first (not materials / products):**
- Spec is tight: sku, name, kind, widthMm, heightMm, priceCents, meta. No open-ended text.
- Pricing model is a single scalar (`priceCents`). No per-slot pricing to infer.
- No cross-references to other rows (materials reference slugs, products reference material slots). A bad extraction doesn't cascade.
- Wins are concrete — a door catalog literally is a grid of "model | size | price".

## User flow

1. On `/admin/catalog/suppliers/[id]`, admin clicks **"Importeer prijslijst"** (new button).
2. Dialog prompts for file upload (PDF / XLSX / CSV, ≤ 20 MB, ≤ 50 pages).
3. File uploads direct-to-Blob → client POSTs `{ blobUrl, filename }` to `/api/admin/imports/supplier-products`.
4. Server creates an `import_jobs` row with `status='pending'`, kicks off async extraction, returns `jobId`.
5. Admin lands on `/admin/catalog/suppliers/[id]/imports/[jobId]` — shows progress ("pagina 3 / 24, 47 rijen gevonden…") via poll or SSE.
6. When `status='ready'`, the review grid renders: one row per candidate product, every field editable, confidence-flagged cells highlighted, validation errors inline, warnings banner at top.
7. Admin un-checks duds, edits iffy rows, clicks **"Importeer 43 producten"** → `POST /api/admin/suppliers/[id]/products/bulk` with the final list → bulk-create in a transaction.
8. Redirect back to the supplier detail page with a toast: "43 producten toegevoegd, 3 overgeslagen".

**Key UX rule:** the extraction job *never* auto-creates rows. The admin is always the last gate. The LLM drafts; the validator filters; the human signs off.

## Data model

One new table:

```
import_jobs                              -- new
  id uuid PK
  tenant_id text NOT NULL FK tenants ON DELETE CASCADE
  supplier_id uuid NOT NULL FK suppliers ON DELETE CASCADE
  created_by text NOT NULL FK user ON DELETE SET NULL  -- audit trail
  kind text NOT NULL                     -- 'supplier-products' (future: 'materials', 'products')
  source_file_url text NOT NULL          -- Blob URL (temp)
  source_filename text NOT NULL          -- original name, for UI
  status text NOT NULL                   -- 'pending' | 'extracting' | 'ready' | 'failed' | 'imported' | 'cancelled'
  status_detail text                     -- progress message or failure reason
  raw_result jsonb                       -- full LLM output (candidates + warnings + metadata)
  model_used text                        -- 'claude-sonnet-4-6' etc.
  cost_cents integer                     -- extraction cost, for reporting
  page_count integer
  candidate_count integer
  imported_count integer                 -- filled at import time
  imported_ids jsonb                     -- [{ sku, id }] for audit
  created_at, updated_at, completed_at
  INDEX (tenant_id, supplier_id, status)
```

No changes to `supplier_products` — bulk-create reuses the existing validator + insert path. The `import_jobs` row is the audit breadcrumb, not a new product container.

**TTL on temp Blobs:** `imports/<tenantId>/<jobId>/<filename>` — swept after 14 days by a weekly cron (new, future work; out of scope for v1).

## API surface

```
POST   /api/admin/imports/supplier-products
  Body: { supplierId, blobUrl, filename }
  Auth: requireBusiness(['super_admin','tenant_admin']) + requireTenantScope
  Returns: { jobId } — 201 immediately; extraction runs async.

GET    /api/admin/import-jobs/[jobId]
  Returns: the full import_jobs row + candidate rows (materialised from raw_result + fresh validation).

POST   /api/admin/import-jobs/[jobId]/cancel
  Sets status='cancelled'. Fire-and-forget; the running extraction finishes and writes to a cancelled job.

POST   /api/admin/suppliers/[id]/products/bulk
  Body: { products: [candidate, …], jobId? }
  Auth: same.
  Inserts in a single transaction. Returns: { inserted: [{ sku, id }], skipped: [{ sku, reason }] }.
  If jobId provided, updates import_jobs.imported_count + imported_ids.
```

The existing `POST /api/admin/suppliers/[id]/products` single-create route stays for the manual flow.

## Extraction architecture

**Server pipeline** on `POST /imports/supplier-products`:

1. Accept the Blob URL. Validate tenant scope + supplier scope.
2. Insert `import_jobs` row with `status='pending'`. Return `jobId`.
3. Fire async extraction (Vercel background function / platform equivalent — not blocking the request):
   - Download the file from Blob (or pass the URL to Claude directly for PDFs).
   - Update job → `status='extracting'`.
   - **Dispatch per file type:**
     - **PDF:** Anthropic Messages API with `document` content block (Claude handles PDFs natively).
     - **XLSX / CSV:** parse locally with `xlsx` / `papaparse`, then a tiny LLM call (`map_columns(headers) → { sku: 'col_B', widthMm: 'col_D', … }`) and direct row mapping. Reserves the heavy extraction budget for real PDFs.
   - Call Claude via `tool_use` with the `extract_supplier_products` tool (schema below).
   - Run each returned candidate through `validateSupplierProductCreate` from `@/domain/supplier`.
   - Compute per-row `status`: `valid` | `invalid` | `duplicate` (existing `(tenantId, kind, sku)` match) | `low_confidence`.
   - Write `raw_result`, `candidate_count`, `cost_cents`, `model_used` → `status='ready'`.
4. Client polls `GET /import-jobs/[jobId]` until `status ∈ {ready, failed, cancelled}`.

**Model choice:**
- Default: **Claude Sonnet 4.6** — best PDF-vision quality-to-cost ratio. Anthropic's native PDF handling is stronger than pre-parsing with `pdf-parse` + text-only.
- Escalation: **Claude Opus 4.7** behind a per-tenant flag for tough scanned catalogs. Triple the cost; only worth it if Sonnet accuracy drops below usable.

**Prompt caching:** the system prompt (schema, extraction rules, examples) is stable across imports and cacheable. On a tenant running 20 imports/month the second onward imports run ~90% cheaper on the input side. Do not cache the document itself.

## Extraction tool schema

```ts
// Tool definition passed to Anthropic
{
  name: 'extract_supplier_products',
  description: 'Extract one product per distinct row from the supplier catalog.',
  input_schema: {
    type: 'object',
    properties: {
      products: {
        type: 'array',
        items: {
          type: 'object',
          required: ['sku', 'name', 'kind', 'widthMm', 'heightMm', 'priceCents'],
          properties: {
            sku: { type: 'string', minLength: 1 },
            name: { type: 'string', minLength: 1 },
            kind: { enum: ['door', 'window'] },
            widthMm: { type: 'integer', minimum: 100, maximum: 5000 },
            heightMm: { type: 'integer', minimum: 100, maximum: 5000 },
            priceCents: { type: 'integer', minimum: 0, maximum: 10_000_00 },
            meta: {
              type: 'object',
              properties: {
                swingDirection: { enum: ['inward', 'outward', 'none'] },
                lockType: { enum: ['cylinder', 'multipoint', 'none'] },
                glazing: { enum: ['solid', 'glass-panel', 'half-glass'] },
                glazingType: { enum: ['double', 'triple', 'single'] },
                uValue: { type: 'number' },
                rValue: { type: 'number' },
                frameMaterial: { type: 'string' },
                openable: { type: 'boolean' },
                leadTimeDays: { type: 'integer' },
              },
            },
            source: {
              type: 'object',
              properties: { pageNumber: { type: 'integer' }, line: { type: 'string' } },
            },
            confidence: {
              type: 'object',
              properties: {
                sku: { type: 'number', minimum: 0, maximum: 1 },
                name: { type: 'number', minimum: 0, maximum: 1 },
                dimensions: { type: 'number', minimum: 0, maximum: 1 },
                price: { type: 'number', minimum: 0, maximum: 1 },
                kind: { type: 'number', minimum: 0, maximum: 1 },
              },
            },
            warnings: { type: 'array', items: { type: 'string' } },
          },
        },
      },
      documentWarnings: { type: 'array', items: { type: 'string' } },
    },
    required: ['products'],
  },
}
```

System prompt (sketch, cached):

```
Role: you extract product rows from supplier price lists for a Belgian configurator
selling garden structures.

For each distinct product line, emit one row via the extract_supplier_products tool.

Rules:
- sku is the supplier's article number; must be non-empty. NEVER invent one.
- name is the human-readable model. Prefer Dutch if present.
- kind is 'door' for doors, 'window' for windows. Skip hardware, locks, hinges, handles,
  gutters, accessories. If ambiguous, pick the best guess and set confidence.kind ≤ 0.6.
- widthMm / heightMm as INTEGERS in millimetres. Convert from cm/m/inch and add a
  warning stating the original unit.
- priceCents in EUR. If the document is USD/GBP, convert at the rate noted in the doc
  or add a warning "currency unclear" + best guess.
- meta fields ONLY if explicitly stated. Do not infer glazing from the product name alone.
- confidence per field: 1.0 if verbatim; 0.6 if inferred from context; 0.3 if guessed.
- source.pageNumber: the page the row came from, for admin citation.

DO NOT invent data. If a required field is missing, omit the row and add a
documentWarning instead.

If the document contains both doors AND windows, emit both — don't pick one.

If bulk pricing is given (10+ €X, 50+ €Y), use the single-unit price.
```

## Safety rails

1. **Validator gate.** Every candidate row runs through `validateSupplierProductCreate` before the UI. Invalid rows render red; can't be imported.
2. **Price-sanity clamp.** Server-side, before the validator: reject `priceCents ≤ 0` or `> €10 000`. Catches unit-confusion errors (`€12` intended as `€1200`) that the LLM might miss.
3. **Duplicate-SKU check.** Before showing the grid, query existing `supplier_products.sku` for the same `(tenantId, kind)`. Matches → render with a banner: "SKU exists — skip / rename / overwrite?". Overwrite is a future polish; v1 offers skip + rename.
4. **Confidence threshold.** Rows with any field confidence < 0.5 → pre-unchecked in the grid. Admin has to tick to import.
5. **Transactional bulk insert.** All-or-nothing. A single slug collision doesn't leave 43 of 44 rows imported — either the whole batch lands or nothing does, and the UI gets the precise offender.
6. **Rate limit.** One active `status ∈ {pending, extracting}` job per `(tenantId, supplierId)`. Prevents accidental double-submits.
7. **Tenant-scoped Blob paths.** `imports/<tenantId>/<jobId>/<filename>` — same pattern as existing `textures/` / `supplier/` uploads, same guard in the upload endpoint.
8. **Audit trail.** `import_jobs.raw_result`, `created_by`, `imported_ids` let us reconstruct what the LLM said, what the admin approved, and which DB rows came from it — useful when a tenant asks "why is this product wrong?".

## Review UI

New component: `BulkImportReviewGrid`.

- Header row: checkbox (select all), SKU, naam, kind, breedte, hoogte, prijs, warnings, bron.
- Row state: `include` checkbox (default: checked if valid + confidence ≥ 0.5, else unchecked).
- Inline edit: every cell becomes editable on click (shadcn `Input` / `Select` matching the supplier-product form).
- Validation badges: red icon + tooltip for validator errors, yellow for low confidence, grey for duplicate SKU.
- "Bron" column: page number + first ~30 chars of the original line (from `source.line` if available) — helps admin cross-check suspicious rows against the source PDF (link opens Blob URL in new tab).
- Bottom bar: `"43 van 50 geselecteerd · Importeer geselecteerde"` button.
- Empty state: "Geen producten gevonden. Controleer het bestand." with a link to documentWarnings.

All shadcn primitives in native form. @dnd-kit not needed — this table doesn't reorder.

## Cost

Per extraction, rough:

| Doc | Tokens in | Tokens out | Sonnet cost | Cached cost |
|---|---|---|---|---|
| 10-page PDF | ~20K | ~2K | $0.09 | $0.04 |
| 20-page PDF | ~40K | ~4K | $0.18 | $0.08 |
| 50-page PDF (cap) | ~100K | ~8K | $0.42 | $0.14 |

At 200 imports/month across all tenants (a generous estimate for v1): <$50/mo in total. Record `cost_cents` on each job so Assymo can build a per-tenant cost report later if needed (and eventually bill for it).

## Rollout

Behind a per-tenant feature flag (new: `tenants.flags jsonb` with `{ aiCatalogIngest?: true }`).

1. Enable for Assymo tenant only. Run one real supplier's catalog end-to-end. Tune the prompt against the output.
2. Enable for the next tenant when they onboard; collect feedback.
3. Remove the flag when the feature is boring.

The flag is also a kill switch — flip off if extraction quality degrades or costs spike.

## Open questions (answer before kickoff)

1. **Background job runtime.** Next.js on Vercel has a 60s Serverless timeout. For 50-page PDFs that's tight. Options:
   - Vercel **Queues** + **background functions** (supports longer runs; preview feature).
   - External queue (Upstash QStash / Inngest).
   - Trigger.dev (clean Anthropic integration).
   - Inline with `Promise.resolve()` (hope Sonnet finishes in 60s; fail over to "try again").

   Decision: start with Vercel background functions (simplest); fall over to Inngest if we hit the wall.

2. **PDF delivery to Anthropic.** Upload to Blob vs. send bytes inline? Anthropic accepts both. Inline is simpler for v1; Blob-URL is cleaner for audit. Go with Blob-URL since we're already uploading there.

3. **XLSX dialect detection.** Tenants will send wildly different column layouts. A small LLM call mapping headers → schema fields seems right. Worth benchmarking against a rule-based approach?

4. **Error UX.** If the LLM returns malformed JSON or the tool call fails, what does the admin see? Probably a "Extractie mislukt. Probeer opnieuw of upload een ander bestand" message with a retry button; log the raw Anthropic response to `import_jobs.raw_result` for debugging.

5. **Update vs overwrite on duplicate SKUs.** v1 treats duplicates as "skip unless admin renames". v2 could offer "update price only" / "update all fields" — useful for re-ingest of an updated catalog.

## Forward-look

This is the first feature in an AI track. Following features should reuse the same pipeline where possible:

- **Phase 6.6 — AI quote explanation.** Reuses `quoteSnapshot`; different model call (pure text, no tools); probably the cheapest next step.
- **Phase 6.7 — Natural-language configuration.** Tool-use against `useConfigStore` mutations. Needs a translation layer from store action names to tool definitions.
- **Phase 6.8 — Product description generator.** A single-call extension to `ProductForm` / `SupplierProductForm` — no new infra.

All of these depend on the same env var (`ANTHROPIC_API_KEY`) and the same per-tenant flag pattern established here.

---

## Not in scope

- Material or product ingest (different shape, higher stakes).
- Re-ingest of an updated catalog (overwrite / diff).
- Image extraction (extracting product photos from the PDF).
- AI-generated hero images for products without photos (separate feature, separate model).
- Multi-supplier detection in one file.
- Tenant self-service billing for AI costs (aggregate `import_jobs.cost_cents` later if needed).
