# Phase 5.8.2 — Poort ProductKind + landing card + deep-link hydration

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `'poort'` a first-class `ProductKind` so tenants can publish gate starter kits ("Standaard Poort 3m enkel", "Poort 4m dubbel gemotoriseerd") as cards on `/`. Clicking a card deep-links into `/configurator?product=<slug>` and hydrates one pre-configured gate from the product's `defaults`. Admin manages these alongside existing overkapping/berging products.

**Architecture:** Phase 5.8.1 shipped the gate primitive end-to-end. This phase wraps it in the existing product-catalog machinery from Phase 5.5.2: extend `ProductKind` enum, extend `ProductDefaults`/`ProductConstraints` jsonb with gate-shaped fields, branch `validateProductCreate/Patch` and `applyProductDefaults` on the new kind, and add a kind-conditional sub-form to `ProductForm.tsx` (mirroring how 5.7's `SupplierProductForm` branches on door vs window). Landing-page card grid + deep-link routing already work for any product kind — no new routing, no new API, no new pages. The admin product form is the only meaningful UI surface added.

**Tech Stack:** Next.js 16 (App Router) + React 19 + Drizzle ORM (Neon HTTP) + shadcn (native) + Vitest via `vite-plus/test`. Package manager: **pnpm**.

**Spec:** Parent design doc `docs/superpowers/plans/2026-04-28-phase-5.8-poorten-design.md` covers Phase 5.8 in full; this plan covers 5.8.2 only.

**Prereqs (operational, not code):**

## Architectural decisions

### Reuse the product machinery, don't fork it

Phase 5.5.2 built `products` as a tenant-scoped table with `kind: text`, jsonb `defaults` + `constraints`, hero image, base price, sort order, soft-delete, and an admin CRUD UI at `/admin/catalog/products`. Phase 5.8.2 plugs `'poort'` into that machinery — same table, same routes, same form file (extended with a kind-conditional branch), same landing page (`/` already iterates `/api/shop/products` regardless of kind). The only architectural call is *where to branch on kind in the form*, and the answer is "the same way 5.7's SupplierProductForm branches on door vs window".

### `ProductDefaults` and `ProductConstraints` extend with optional gate-shaped fields

Today both jsonb shapes are tuned for structural buildings (width/depth/height + per-slot material defaults; min/max dims + per-slot allow-lists). Adding `'poort'` doesn't fit those slot semantics — gates have no walls/roof/floor and one material slot ("the leaf"). Two design options:

- **(a) Add a `poort: { … }` discriminated subobject** to `ProductDefaults` and `ProductConstraints`. Existing structural fields stay typed for overkapping/berging; the new subobject carries gate fields exclusively.
- **(b) Widen `ProductDefaults` with optional flat fields** (`partCount?`, `partWidthMm?`, `swingDirection?`, etc.) that only apply when `kind === 'poort'`.

**Choose (a)** — discriminated subobjects keep types tight (TS narrows on `kind`) and avoid 7+ optional sibling fields that are nonsense for non-poort kinds. Mirrors the design intent: gate is a different shape of product.

```ts
type ProductDefaults = {
  // existing structural fields (overkapping / berging) stay where they are
  width?: number
  depth?: number
  height?: number
  materials?: { wallCladding?: string, roofCovering?: string, … }

  // new — only consulted when kind === 'poort'
  poort?: {
    partCount?: 1 | 2
    partWidthMm?: number
    heightMm?: number
    swingDirection?: 'inward' | 'outward' | 'sliding'
    motorized?: boolean
    materialId?: string
  }
}

type ProductConstraints = {
  // existing
  minWidth?, maxWidth?, …
  allowedMaterialsBySlot?: Partial<Record<ProductSlot, string[]>>

  // new
  poort?: {
    partCountAllowed?: (1 | 2)[]
    partWidthMinMm?: number
    partWidthMaxMm?: number
    heightMinMm?: number
    heightMaxMm?: number
    swingsAllowed?: ('inward' | 'outward' | 'sliding')[]
    motorizedAllowed?: boolean
    allowedMaterialSlugs?: string[]   // filters the gate-catalog picker
  }
}
```

This keeps existing overkapping/berging products' jsonb untouched and additive at the type level.

### `validateProductCreate/Patch` branches on kind

The validator already has a structural-only path. Add a `'poort'` branch that:

- Rejects if the tenant has zero `'gate'` materials (`materials.filter(m => m.categories.includes('gate') && !m.archivedAt).length === 0`). Stable error code: `product_kind_unsupported_for_tenant` (or whatever the existing convention is — match exactly).
- Validates `defaults.poort.partCount ∈ {1, 2}` if present.
- Validates `defaults.poort.partWidthMm` and `heightMm` against `getConstraints('poort')` from `src/domain/building/constants.ts` (Task 1 of 5.8.1 added these).
- Validates `defaults.poort.swingDirection ∈ {'inward','outward','sliding'}` if present.
- Validates `defaults.poort.materialId` is the slug of an existing gate-category material (or empty).
- Validates `constraints.poort.allowedMaterialSlugs` references real gate-category materials.
- Validates `constraints.poort.partWidthMinMm/MaxMm` and `heightMinMm/MaxMm` are within the building-constraint envelope (`getConstraints('poort')` is the global hard limit).
- Rejects gate fields nested under `defaults.poort` / `constraints.poort` when `kind !== 'poort'` (cross-kind contamination guard).

### `applyProductDefaults` extends to hydrate poort

Today `applyProductDefaults(product, materials)` returns a `BuildingEntity` for `'overkapping'` or `'berging'`. Extend it so `kind === 'poort'` returns a `BuildingEntity` of `type: 'poort'` whose `gateConfig` is built from `product.defaults.poort` (falling back to `defaultGateConfig()` for any missing field). Reuses Task 1's `createGateBuildingEntity` factory for the actual entity construction.

```ts
case 'poort': {
  const d = product.defaults?.poort ?? {}
  return createGateBuildingEntity({
    gateConfig: {
      partCount: d.partCount ?? defaultGateConfig().partCount,
      partWidthMm: d.partWidthMm ?? defaultGateConfig().partWidthMm,
      heightMm: d.heightMm ?? defaultGateConfig().heightMm,
      swingDirection: d.swingDirection ?? defaultGateConfig().swingDirection,
      motorized: d.motorized ?? defaultGateConfig().motorized,
      materialId: d.materialId ?? '',
    },
  })
}
```

### Admin UI: one new sub-form, no new pages

`ProductForm.tsx` already has a `kind` selector (radio or select) and conditionally renders structural-vs-poort sections. Add `<GateProductDefaults>` as a kind-conditional sub-component that renders only when `kind === 'poort'`. Inside it:

- Parts: shadcn `ToggleGroup` (1 / 2) for `defaults.poort.partCount`. Allow-list-aware: greyed out values not in `constraints.poort.partCountAllowed` if set.
- Per-part width: numeric input (m or mm — match `MaterialForm`'s convention; gate sidebar uses meters). Bound to `defaults.poort.partWidthMm`. Two min/max constraint inputs for `constraints.poort.partWidthMinMm/MaxMm` directly below.
- Height: same pattern as width. Two constraint inputs for `heightMinMm/MaxMm`.
- Default swing: shadcn `ToggleGroup` for `defaults.poort.swingDirection`. Multi-check sibling for `constraints.poort.swingsAllowed`.
- Motorized: switch for `defaults.poort.motorized`. Tri-state checkbox for `constraints.poort.motorizedAllowed` (allowed / disallowed / not specified).
- Default material: shadcn `Select` populated from the tenant's gate catalog (filtered by `constraints.poort.allowedMaterialSlugs` if set). Multi-select sibling for the constraint allow-list.

All controls dispatch through the existing form's `onSubmit` body that PATCHes the jsonb. No new API endpoints.

### Landing-page card + deep-link hydration

Both already work. `/` reads `/api/shop/products` and renders any kind as a card; the existing `<ProductCard>` shows hero image + name + base price (or "Op aanvraag" if `basePriceCents === 0`) + a "Configureer" CTA. Clicking the card deep-links to `/configurator?product=<slug>`, which hydrates the configurator's first-mount via `applyProductDefaults`. Once Task 4 of this phase extends `applyProductDefaults` for poort, the deep link works automatically.

## Notes on modularity

This phase is small precisely because Phase 5.8.1 paid the architectural cost upfront. The product-catalog machinery from Phase 5.5.2 (admin CRUD, landing page, deep-link hydration) treats kinds as pluggable; adding a third kind is an enum value + jsonb shape + form branch + applyProductDefaults branch. No table changes, no API changes, no new pages, no new routing.

The same pattern works for any future ProductKind. If Phase 5.9 wanted to add `'poolhuis'` as a starter-kit type, the work would be the same shape: register the BuildingType (5.8.1 equivalent), then wrap it as a Product (5.8.2 equivalent). The 5.8 series demonstrates the pattern end-to-end.
