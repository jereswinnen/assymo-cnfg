# Phase 5.8.3 — Gate SupplierProductKind + per-instance SKU/options picker

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `'gate'` a first-class `SupplierProductKind` so tenants can publish branded gate models (with option matrices for color / lock / handle / glazing / motor) alongside doors and windows. Customers pick a SKU + options on a placed gate via a new dropdown block in the existing `GateConfigPanel`. Pricing layers supplier-base + selected option surcharges + motor surcharge on top of the per-m² + price-book stack already shipped in 5.8.1. Order submit freezes the SKU and the customer's selected options so historical orders stay stable against catalog drift.

**Architecture:** Phase 5.7 shipped polymorphic `supplier_products` (`kind: 'door' | 'window'`, `meta: jsonb`) with a per-tenant catalog, an admin CRUD form that branches on kind, a public `GET /api/shop/supplier-products?kind=…` endpoint, a placement validator on order submit, and a quote-snapshot freeze step that copies the matched product row onto each supplier-driven line item. Phase 5.8.3 plugs `'gate'` into that machinery: extend the kind enum, add `GateMeta` + `validateGateMeta`, branch `SupplierProductForm.tsx` with a new `<GateMetaForm>` sub-component, extend the placement validator with a poort-buildings branch (gates aren't openings on walls — they're standalone primitives whose chosen dimensions must fit the SKU's `maxDimensions`), extend the gate pricing helper to consume supplier-base + option surcharges when `supplierProductId` is set, extend the snapshot freeze step to cover gate-driven line items, and add a supplier-product picker block to the existing `GateConfigPanel.tsx`. No new tables, no new admin pages, no new API routes.

**Tech Stack:** Next.js 16 (App Router) + React 19 + Drizzle ORM (Neon HTTP) + shadcn (native) + Vitest via `vite-plus/test`. Package manager: **pnpm**.

**Spec:** Parent design doc `docs/superpowers/plans/2026-04-28-phase-5.8-poorten-design.md` covers Phase 5.8 in full; this plan covers 5.8.3 only.

## Architectural decisions

### Reuse the supplier-product machinery, don't fork it

5.7 built `supplier_products` as a tenant-scoped table with `kind: text`, jsonb `meta`, hero image, dimensions (widthMm + heightMm), base price, sort order, soft-delete, FK supplier_id RESTRICT, and an admin CRUD UI at `/admin/catalog/suppliers/[id]/products` whose form branches on door vs window. 5.8.3 plugs `'gate'` into that machinery — same table, same routes, same form file (extended with a kind-conditional branch), same public read endpoint (already kind-parametric). The architectural call is *where to branch on kind* and the answer is "the same way the door vs window branches work today".

### `GateMeta` is one supplier-product row carrying option matrices

Lock variants, handle SKUs, RAL colors, glazing options, motor models all live as **option arrays on a single `'gate'` SupplierProduct**, not as separate supplier-product kinds. Mirrors how a door SKU today carries `swingDirection` / `lockType` / `glazing` as flat enum fields — gates just need richer per-option matrices because the variation surface is bigger.

```ts
export interface GateMetaOption {
  sku: string;                    // stable id within this product, used in GateConfig.selectedXxxSku
  labelKey?: string;              // optional i18n key; falls back to literal label
  label?: string;                 // free-form display name when labelKey omitted
  ralCode?: string;               // colors only
  surchargeCents?: number;        // additive on the supplier-base price
}

export interface GateMeta {
  partCount?: 1 | 2 | 'configurable';
  motorized?: boolean | 'optional';
  motorizedSurchargeCents?: number;          // applied when motorized==='optional' AND customer ticks the toggle
  swingDirections?: ('inward' | 'outward' | 'sliding')[];
  defaultDimensions?: { widthMm: number; heightMm: number };
  maxDimensions?:     { widthMm: number; heightMm: number };
  glazing?: 'none' | 'partial' | 'full';
  availableColors?:  GateMetaOption[];
  availableLocks?:   GateMetaOption[];
  availableHandles?: GateMetaOption[];
  rValue?: number;
  leadTimeDays?: number;
}
```

All fields optional for tolerance — a minimal viable gate SKU has just a name, dimensions, and a base price; option-matrix richness is incremental authoring.

### Placement validation: gates aren't openings on walls

Doors + windows live as openings on a `WallConfig`; the existing `validateSupplierPlacements` walks `building.walls` and checks each opening fits the wall. Gates are *standalone `BuildingEntity`s* of `type='poort'` whose footprint comes from `building.dimensions` + `building.heightOverride ?? defaultHeight`. The validator gains a parallel branch:

```ts
// Iterate poort buildings
for (const building of buildings) {
  if (building.type !== 'poort') continue;
  const cfg = building.gateConfig;
  if (!cfg?.supplierProductId) continue;
  const product = productMap.get(cfg.supplierProductId);
  if (!product || product.archivedAt !== null) continue;
  const placedWidthMm  = building.dimensions.width  * 1000;
  const placedHeightMm = (building.heightOverride ?? defaultHeight) * 1000;
  const max = (product.meta as GateMeta).maxDimensions;
  if (max) {
    if (placedHeightMm > max.heightMm) issues.push({ code: 'gate_placement_too_tall', … });
    if (placedWidthMm  > max.widthMm)  issues.push({ code: 'gate_placement_too_wide', … });
  }
}
```

New stable error codes (per the project's "stable error code" convention): `gate_placement_too_tall`, `gate_placement_too_wide`. Existing `supplier_product_not_found` / `supplier_product_archived` cover the existence checks (already triggered by the order route's referenced-id walk — extend that walk to include `building.gateConfig?.supplierProductId`).

### Pricing: supplier-base + option surcharges layer onto the existing gate pricing

`src/domain/pricing/gate.ts` already produces line items for naked gates (per-m² material + per-leaf base + motor + sliding surcharges from `priceBook.poort`). Extend it so when `gateConfig.supplierProductId` is set:

1. **Replace** the per-m² material line with a single `quote.line.gateSupplier` line whose total = `product.priceCents`. The naked-gate per-m² calculation only makes sense for material-only gates; a branded SKU has its own pricing.
2. **Append** option surcharges as separate line items (one per non-zero surcharge): `quote.line.gateOption` with labelParams `{ optionLabel }`. Selected option SKUs come from `gateConfig.selectedColorSku` / `selectedLockSku` / `selectedHandleSku`; their `surchargeCents` is summed into the line.
3. **Motor surcharge**: when `meta.motorized === 'optional'` AND `gateConfig.motorized === true`, add `meta.motorizedSurchargeCents`. When `meta.motorized === true` (forced on), the motor cost is included in the base — no extra line. When `meta.motorized === false` (forced off), the toggle is hidden in the UI and no motor is charged. The existing `priceBook.poort.motorSurchargeCents` only applies to naked gates.
4. **Sliding** + **per-leaf** surcharges from `priceBook.poort` continue to apply when the placed gate is sliding / 2-part — those are tenant-level dials independent of the SKU.

The `LineItemSource` for SKU-driven gate lines becomes `{ kind: 'supplierProduct'; productId: string }` (same shape doors use today) so the existing snapshot freeze step picks them up automatically. Naked-gate lines keep `kind: 'gate'`.

### Snapshot: freeze the chosen options into the line item

The supplier-product freeze step in `buildQuoteSnapshot` already attaches `supplierProduct` to any line item whose `source.kind === 'supplierProduct'`. By switching SKU-driven gate lines to `kind: 'supplierProduct'`, the freeze step covers them with no change.

What's *new* is freezing the customer's *selected option SKUs*. The selected color/lock/handle is part of the `GateConfig`, which `buildConfigSnapshot` already deep-clones. But the human-readable labels for those options come from `meta.availableColors[].labelKey` (or `.label`). To survive a future supplier-product edit that renames or removes an option, the order needs the *resolved* option labels at submit time. Two options:

- **(a) Snapshot raw + resolve at render**: keep config & quote as-is, attach the full meta (already inside `frozen.supplierProduct.meta`), and resolve `selectedColorSku` → label at render time using the frozen meta. Already covered by the existing freeze logic.
- **(b) Pre-resolve labels into per-option line items**: emit one line item per selected option with a frozen `optionLabel` string in `labelParams`. Resilient against meta drift but adds line-item bulk.

**Choose (a)** — the freeze step already captures `meta` verbatim, so `frozen.supplierProduct.meta.availableColors` survives any future edit. Resolving the label at render time costs one map lookup; the alternative duplicates the data.

### Admin UI: one new sub-form, no new pages

`SupplierProductForm.tsx` already branches on `kind` (`'door'` vs `'window'`) — the third option drops in alongside. Touch points:

- **Kind selector** at the top of the form: add the third option `"Poort"` with value `'gate'`.
- **`<GateMetaForm>` sub-component** (new file `src/components/admin/catalog/GateMetaForm.tsx`), kind-conditional render, parallels the existing `kind === 'door'` and `kind === 'window'` blocks. Inside it:
  - Parts: shadcn `ToggleGroup` (1 / 2 / Vrij) bound to `meta.partCount`.
  - Motorized: tri-state radio (Altijd / Nooit / Optioneel) bound to `meta.motorized`. When "Optioneel" is selected, reveal the `motorizedSurchargeCents` numeric input.
  - Swing directions: 3 checkboxes (Naar binnen / Naar buiten / Schuivend) bound to `meta.swingDirections[]`.
  - Default dimensions: two numeric inputs (mm) bound to `meta.defaultDimensions.widthMm` + `.heightMm`.
  - Max dimensions: two numeric inputs (mm) bound to `meta.maxDimensions.widthMm` + `.heightMm`.
  - Glazing: shadcn `RadioGroup` (Geen / Gedeeltelijk / Volledig) bound to `meta.glazing`.
  - Available colors / locks / handles: three repeating sub-form blocks (one shadcn `Card` per option, "+ optie toevoegen" button at the bottom of each). Each option row has SKU, label or i18n key, optional RAL code (colors only), surchargeCents.
  - Lead time + R-value: numeric inputs (parallel to door/window).
- **Submit body** in the form's existing PATCH/POST handler: serialise the `meta` jsonb when `kind === 'gate'`. Existing form-state plumbing already passes whatever `meta` shape is set.

The product-form-schema (zod or `validateSupplierProductCreate/Patch` runtime call) absorbs the new `meta` shape via `validateGateMeta`.

### Configurator UI: one new block in the existing panel

`GateConfigPanel.tsx` already renders parts / material / swing / motorized for a placed gate. Phase 5.8.3 adds a fifth block at the top (above "Aantal delen") — a supplier-product picker — and three conditional option dropdowns.

- **Supplier product**: shadcn `Select` populated from a new selector hook `useTenantSupplierProducts({ kind: 'gate' })` (or extend the existing `useTenantCatalogs` if it already loads supplier products — recon during execution). Empty option = "Naakt (eigen materiaal)" → naked-gate flow, current behaviour. Selecting a SKU sets `gateConfig.supplierProductId` and applies the SKU's `defaultDimensions` to `building.dimensions` if present.
- **When a SKU is selected**, conditionally render four extra controls:
  - Color: shadcn `Select` from `meta.availableColors` (skip if empty).
  - Lock: shadcn `Select` from `meta.availableLocks` (skip if empty).
  - Handle: shadcn `Select` from `meta.availableHandles` (skip if empty).
  - Motor: keep the existing `Switch` but flip its enabled-state from `meta.motorized` (`true` forces on + disabled, `false` forces off + disabled, `'optional'` allows toggle).
- **When a SKU is selected**, the existing Parts toggle, Swing toggle, and Material picker get filtered/disabled according to the SKU's meta (`partCount`, `swingDirections`, no material picker because the SKU's hero image / textures own the visual).

All controls dispatch through the existing `updateGateConfig(buildingId, patch)` mutation. New `GateConfig` fields:

```ts
export interface GateConfig {
  // existing
  partCount: 1 | 2;
  materialId: string;          // still valid for naked gates; ignored when supplierProductId set
  swingDirection: GateSwingDirection;
  motorized: boolean;
  // new (Phase 5.8.3)
  supplierProductId?: string | null;
  selectedColorSku?: string | null;
  selectedLockSku?: string | null;
  selectedHandleSku?: string | null;
}
```

`canonicalizeConfig` keeps these fields in stable order (parallel to `WallConfig.doorSupplierProductId`) so content-hash dedup stays deterministic. `migrateConfig` is a no-op for legacy gates (the new fields are optional).

### Public API: extend the existing kind-parametric endpoint

`GET /api/shop/supplier-products?kind=door|window` already takes `kind` as a query param. The route enum gains `'gate'`. No code change beyond expanding the validator's allowed-kind set; the existing host-scoped, archived-supplier-filtered query body already returns `meta` jsonb verbatim and the client-side picker hook consumes it.

### Order submit: existence + placement validation

`POST /api/shop/orders` already walks `building.walls` to collect referenced supplier-product IDs. Extend the walk to also include `building.gateConfig?.supplierProductId`. The existence/archive check is identical for the new IDs. The placement check gains the poort-buildings branch described above.

### No DB migration

`supplier_products.kind` is text, validated in code. `supplier_products.meta` is jsonb. Adding `'gate'` is a code-only change; `GateMeta` rides along in jsonb.

### Out of scope (deferred or never)

- Multi-leaf gates beyond 2 parts.
- Animated gate open/close in 3D (closed-state render only — same as 5.8.1).
- Sliding-gate rail rendering in 3D.
- Per-supplier feature flags or admin-side guard rails on which kinds a supplier can publish.
- Replacing the 3D material textures with a flat hero-image when a SKU is selected (visual upgrade — track separately if a tenant pushes back on the look).
- Bundled hardware kits (a separate "kit" SKU referencing multiple option SKUs). Today every option is à la carte.
- Pricing currency mixing (all option surcharges share the supplier product's currency, which already shares the tenant's currency).

## Pricing changes

`src/domain/pricing/gate.ts` extension. New branch at the top:

```ts
const sku = gateConfig.supplierProductId
  ? supplierProducts.find(p => p.id === gateConfig.supplierProductId && p.archivedAt === null)
  : null;

if (sku) {
  const meta = sku.meta as GateMeta;
  // Base
  items.push({
    labelKey: 'quote.line.gateSupplier',
    labelParams: { name: sku.name },
    total: sku.priceCents / 100,
    source: { kind: 'supplierProduct', productId: sku.id },
  });
  // Selected option surcharges
  for (const [field, list] of [
    ['selectedColorSku',  meta.availableColors],
    ['selectedLockSku',   meta.availableLocks],
    ['selectedHandleSku', meta.availableHandles],
  ] as const) {
    const selectedSku = gateConfig[field];
    if (!selectedSku || !list) continue;
    const opt = list.find(o => o.sku === selectedSku);
    if (!opt?.surchargeCents) continue;
    items.push({
      labelKey: 'quote.line.gateOption',
      labelParams: { name: opt.label ?? '', labelKey: opt.labelKey },
      total: opt.surchargeCents / 100,
      source: { kind: 'supplierProduct', productId: sku.id },
    });
  }
  // Optional-motor surcharge
  if (meta.motorized === 'optional' && gateConfig.motorized && meta.motorizedSurchargeCents) {
    items.push({ labelKey: 'quote.line.gateMotor',
                 total: meta.motorizedSurchargeCents / 100,
                 source: { kind: 'supplierProduct', productId: sku.id } });
  }
  // Sliding + per-leaf surcharges still apply (priceBook-level)
  if (priceBook.poort?.slidingSurchargeCents && gateConfig.swingDirection === 'sliding') {
    items.push({ labelKey: 'quote.line.gateSliding',
                 total: priceBook.poort.slidingSurchargeCents / 100,
                 source: { kind: 'supplierProduct', productId: sku.id } });
  }
  if (priceBook.poort?.perLeafBaseCents) {
    items.push({ labelKey: 'quote.line.gatePerLeaf',
                 labelParams: { count: gateConfig.partCount },
                 total: (gateConfig.partCount * priceBook.poort.perLeafBaseCents) / 100,
                 source: { kind: 'supplierProduct', productId: sku.id } });
  }
} else {
  // existing naked-gate path: per-m² + priceBook surcharges with kind: 'gate' source
}
```

Tests in `tests/pricing-gate-supplier.test.ts` cover: SKU base + multi-option pricing, motor-optional toggle on/off, sliding still applies, per-leaf still applies, missing SKU falls back to naked path, archived SKU falls back to naked path, option SKU not in meta is silently dropped (defensive), kind-agnostic snapshot freeze captures all gate lines.

## Notes on modularity

This phase is small precisely because Phases 5.7 and 5.8.1/5.8.2 paid the architectural cost upfront. The supplier-product machinery from 5.7 (admin CRUD, public read endpoint, placement validator, snapshot freeze) treats kinds as pluggable; the kind-agnostic freeze step in particular is what lets gate-with-SKU lines piggy-back on the existing door/window snapshot path with zero new freeze code. The poort-buildings branch in `validateSupplierPlacements` is the only meaningful structural addition — and that's because gates *are* primitives, not openings.

The same pattern works for any future SupplierProductKind. If a future phase wanted to add `'shutter'` or `'skylight'`, the work would be the same shape: register the kind, add a `Meta` struct + validator, add the placement-fit branch (on whichever `BuildingEntity` shape it lives on), add an admin form sub-component, add a configurator picker block. No table changes, no new API routes, no new admin pages.
